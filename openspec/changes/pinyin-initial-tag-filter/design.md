## Context

`useTagAutocomplete.ts` 中的标签过滤目前使用 `tag.name.toLowerCase().includes(query)` 进行纯文本匹配。中文标签需要用户切换输入法输入汉字才能匹配，体验不佳。

## Goals / Non-Goals

**Goals:**
- 用户输入拼音首字母即可匹配中文标签（如输入 `sj` 匹配「数据」）
- 保持现有文本匹配功能不变
- 零后端改动

**Non-Goals:**
- 不支持全拼匹配（如 `shuju`）
- 不支持模糊音（如 `zh`/`z` 不区分）
- 不对标签做预索引或缓存拼音映射

## Decisions

### 1. 拼音库选择：使用静态汉字→首字母映射表

**选择**：内嵌一个轻量的汉字→拼音首字母映射（约 6800 个常用汉字，压缩后 ~15KB），不引入外部 npm 依赖。

**备选方案**：
- `pinyin-pro`：功能完整但体积大（~100KB+），对本需求过重
- `tiny-pinyin`：较轻量但仍是外部依赖

**理由**：本功能只需要首字母，不需要完整拼音。一个静态映射表足够，且避免了依赖管理和打包体积问题。

### 2. 匹配策略：拼音首字母前缀匹配

对标签名中每个中文字符提取首字母拼接成字符串，用户输入作为前缀匹配。混合文本（中英混合标签）中英文字符保留原样参与匹配。

示例：「数据分析」→ `sjfx`，输入 `sj` 匹配成功。

### 3. 排序优先级

精确文本匹配排在拼音匹配前面，拼音匹配内部按最近使用排序（复用现有逻辑）。

## Risks / Trade-offs

- **多音字**：静态映射表每个汉字只取一个读音，可能导致部分多音字标签无法通过次要读音匹配 → 可接受，首字母匹配是辅助手段，用户也可用汉字直接匹配
- **生僻字缺失**：映射表可能不包含极生僻汉字 → 回退到普通文本匹配，不影响使用
