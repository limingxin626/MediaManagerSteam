## Context

MediaManagerSteam 是一个自托管的个人媒体管理应用，使用 FastAPI + SQLite 后端、Vue 3 前端。当前没有数据库管理界面，数据维护需要直接操作 SQLite 文件。项目LAN-only、无认证。

数据库包含 6 张表：Message、Media、Actor、Tag、message_media（关联表）、message_tag（关联表）、sync_log。现有 API 已提供各实体的 CRUD 端点。

## Goals / Non-Goals

**Goals:**
- 提供数据库概览仪表盘（各表记录数、最近活动）
- 提供通用表浏览器：分页列表、搜索、记录详情查看与内联编辑
- 支持删除记录（带确认）
- 集成到现有导航体系中
- 复用现有 API 端点，最小化后端改动

**Non-Goals:**
- 不做通用 SQL 查询执行器（安全风险）
- 不做数据库结构迁移工具
- 不做批量导入/导出功能
- 不做数据备份/恢复功能
- 不做移动端适配（管理页面仅桌面使用）

## Decisions

### 1. 前端纯路由方案 vs 独立应用

**选择：前端纯路由**。在现有 Vue 应用中新增 `/admin` 路由，而非独立部署一个管理应用。

理由：项目是个人工具，用户上下文一致。新增一个路由远比维护独立应用简单。共享 useApi、useToast 等 composable。

### 2. 复用现有 API vs 新建通用 CRUD API

**选择：混合方案**。对已有 CRUD 端点的实体（Message、Media、Actor、Tag）复用现有 API。新增一个 `/admin/stats` 端点提供统计数据。对 sync_log 等缺少 API 的表，新增简单的查询端点。

理由：避免重复创建 API。现有端点已经被验证，只需在前端适配展示格式。仅在缺口处补充。

### 3. 表格组件：自建 vs 第三方库

**选择：自建轻量表格组件**。使用 Tailwind 样式的 `<table>` + 自定义分页组件。

理由：项目仅 6 张表、字段有限，不需要 AG Grid 等重量级库。自建可完全控制样式一致性，减少依赖。与项目已有的 Tailwind + HeadlessUI 风格一致。

### 4. 编辑模式：弹窗 vs 内联

**选择：侧边抽屉面板**。点击记录行时，右侧滑出详情/编辑面板，表格保持可见。

理由：比弹窗更自然（能看到列表上下文），比内联编辑更不易误操作。HeadlessUI 已有 Dialog/Transition 组件可复用。

### 5. 页面结构

采用 Tab 切换式布局：
- **概览 Tab**：数据库统计卡片（各表记录数、存储空间、最近更新）
- **表浏览 Tab**：左侧表列表，右侧表内容（分页表格 + 搜索 + 筛选）

## Risks / Trade-offs

- **[数据安全]** 管理页面可直接编辑/删除数据 → 所有破坏性操作都加确认弹窗；项目是 LAN-only 无认证，与现有安全模型一致
- **[前端包体积]** 新增管理页面增加代码量 → 使用路由懒加载 (`() => import(...)`)，管理页面不进入主 bundle
- **[维护成本]** 数据模型变更需同步更新管理页表定义 → 表的字段定义集中在一个配置对象中，修改成本低
