# 2026-04-30 开发日志

## 拆分消息后保持滚动位置

**问题**：在 Message 页拆分一条消息时，`handleSplit` 调用 `resetAndFetch({})` 重新拉取整个列表，`fetchMessages(false)` 末尾无条件 `scrollToBottom('auto')`，导致用户视图被强制拉到列表底部，正在查看的位置丢失。

**修复**：`vue/src/views/Message.vue` 的 `handleSplit` 改为局部更新而非整列表 reset：

1. `POST /messages/{id}/split` 接口已直接返回新建 message，无需再次 GET 列表。
2. 同时 `GET /messages/{id}` 拿到原 message 最新详情（媒体减少后的状态）。
3. 在 `messages.value` 中找到原 message index：用 `full` 替换原项，再把 `newMsg` 插入到 `idx + 1`（后端保证 newMsg.created_at 大于原 message，reverse 显示后视觉上紧邻其下方）。
4. 滚动锚定：拆分前记录原 message DOM 的 `getBoundingClientRect().top`，DOM 更新后 `scrollTop += topAfter - topBefore`，原 message 视觉上原地不动，新 message 自然在其下方原地出现。

不再调用 `resetAndFetch` 和 `scrollToBottom`。

**验证**：列表中部、顶部、底部分别拆分一条多媒体消息，视口位置保持不变，新消息出现在原消息下方。

## 修复详情面板进入预览后删除媒体不刷新

**问题**：在 Message 页，从 MessageCard 点击媒体进入 MediaPreview，删除媒体会立即从卡片消失；但是先点击消息打开右侧详情面板，再从详情面板点击媒体进入预览删除，详情面板中的媒体不会消失。

**根因**：`handleMessageClick` 重新 `GET /messages/{id}` 拿到 `full` 赋给 `selectedMessage`，其 `media_items` 与 `messages[idx].media_items` 是不同的数组引用。`handleMediaDeleted` / `handleMediaRotated` / `handleMediaTagsChanged` 只更新了 `messages` 列表中的对象，没有同步 `selectedMessage`。

**修复**：`vue/src/views/Message.vue` 三个 handler 在更新 `messages` 后，若 `selectedMessage.value?.id` 与当前预览的消息一致，再对 `selectedMessage.value.media_items` 做同样操作（删除 splice、旋转添加缓存破坏 query、tags 替换）。删除时同步更新 `media_count`。

## 修复图片 EXIF orientation 导致 mosaic 比例失真

**现象**：MessageCard mosaic 中某些竖屏照片显示比例极端夸张（远超 9:21），左右被裁剪。

**根因**：`backend/app/utils/__init__.py` 的 `MediaInfoUtils.get_media_info` 读取图片尺寸时直接使用 `PIL.Image.open(...).size`，未应用 EXIF orientation。手机竖屏拍摄的 JPEG 通常是 raw 横向像素 + orientation=6/8 旋转标记，导致存入数据库的 width/height 是横向的（如 4000×2252 而非 2252×4000）。前端按错误的 ratio 走 mosaic 布局算法，把竖图当横图处理，同行另一张被 clamp 的竖图就被压成极端窄高。同文件的缩略图生成已正确调用 `ImageOps.exif_transpose`，所以缩略图本身方向是对的——只是数据库尺寸错了。

**修复**：
1. `backend/app/utils/__init__.py:397` IMAGE 分支改为 `ImageOps.exif_transpose(img)` 后再读 `.size`。EXIF 日期/GPS 仍从原 `img` 读取（exif_transpose 会清掉部分 metadata）。
2. 新增一次性脚本 `backend/scripts/fix_image_orientation.py`，遍历所有 image media，重新计算正确的 width/height 写回。

**验证**：运行修复脚本后，原本极端比例的 mosaic 恢复正常布局。

## 修复 mosaic 行容器被图片撑高

**现象**：修复 EXIF orientation 后，某些 mosaic（含两张竖图的行）整体高度仍然异常，F12 看到容器被撑到 726×1103，而 `aspect-ratio` 计算应当是 726×771。

**根因**：`MessageCard.vue` 的 ROWS 布局，外层容器靠 `aspect-ratio` 推高度，行容器用 `flex: heightWeight 1 0%` 按 grow 比例分配。但行容器内的 `<img class="w-full h-full">` 在 flex-grow 收缩时缺少 `min-height: 0`，会按图片内容请求扩展行高，进而把 aspect-ratio 容器整体撑大、grow 比例失真。

**修复**：行容器从 `flex: heightWeight` 改为显式 `height: X%`（基于固定 aspect-ratio 切割），并加 `min-h-0`。新增 `mosaicRowHeights` computed，计算每行的高度百分比。这样 cell 高度严格由容器 aspect-ratio × 行百分比决定，不再受 img 内容影响。

## 修复 Media 页右侧时间轴随滚动位移

**现象**：Media 页右侧的 DateScrubber 时间轴会随着网格滚动一起向上移动，应当固定在视口右侧。

**根因**：`vue/src/views/Media.vue` 中 DateScrubber 用 `absolute right-0 top-0 bottom-0` 定位，但其最近的定位祖先是滚动容器内部的内容流（`.flex-1.overflow-y-auto` 内层），随内容滚动。

**修复**：把 DateScrubber 和"回到最新"按钮移出滚动容器，作为主内容列（`flex-1 flex flex-col min-w-0`）的兄弟节点；该列加 `relative`，使 DateScrubber 的 `absolute` 锚定到不滚动的列容器上，时间轴始终覆盖右侧整列高度。

## Message 顶部标题栏与底部输入框对齐 MessageCard 内容区

**问题**：Message 页顶部的标题/合并/收藏/搜索栏和底部"写点什么"输入框，与中间 MessageCard 列表的视觉宽度不一致——标题栏右侧出现多余空白，底部输入框宽度也对不齐。

**根因**：
1. header 容器原为 `px-3` + `max-w-4xl`，feed 容器为 `px-4 sm:px-6 lg:px-8` + `max-w-3xl`，padding 与 max-width 都不一致。
2. `SearchInput.vue` 根 div 写死了 `flex-1 max-w-md`（448px），即使外层 flex 给的剩余空间更大，搜索框也只撑到 448px，右侧出现空白。
3. MessageCard 根是 `flex items-end gap-2`，右侧有一个 `w-8` 的展开按钮（Telegram 风格箭头），内容区实际只占左侧 `flex-1`。header 和底部输入框若按 `max-w-3xl` 完全对齐外框，会比"内容区"多出 40px（按钮 32px + gap 8px），看起来仍然偏移。

**修复**：`vue/src/views/Message.vue`
1. header 与底部输入框统一为 `px-4 sm:px-6 lg:px-8` + `max-w-3xl mx-auto`，与 feed 一致。
2. header 内层 flex 用 `justify-between`，标题/按钮组靠左、搜索靠右，不再因为 SearchInput 的 `max-w-md` 上限露出空白。
3. header 与底部输入框的内层容器各加 `pr-10`，吃掉 MessageCard 右侧展开按钮 + gap 占用的 40px，让宽度严格对齐内容区。

## Message 左栏：标签/演员各占一半独立滚动

**改动**：`vue/src/views/Message.vue` 左栏原本整列共用一个 `overflow-y-auto`，标签多时演员被挤到下方需要长滚动才能看到。现拆为上下两个 `flex-1 min-h-0` 容器，分别独立 `overflow-y-auto`，中间用 border-t 分隔。标签和演员各占左栏一半高度，互不影响滚动。
