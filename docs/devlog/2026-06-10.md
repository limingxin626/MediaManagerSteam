# 2026-06-10

## fix(mac): 消息页媒体预览改为 overlay 模式 + 容器级焦点管理

### 背景

Mac 端消息页(`MessagesView`)的媒体预览有两个 bug:

1. **滚动位置丢失**:在消息流中向下滚动若干屏,点击某条消息卡的某张媒体进入预览,关闭后消息列表回到顶部 —— 用户必须重新滚到原位置才能继续看。
2. **空格误触发**:关闭预览后再按一次空格,会打开另一张不相关的媒体,或者激活 toolbar 上的星标/刷新按钮、FilterSidebar 里某个 chip。

对照组:媒体库(`MediaLibraryView`)走的是同样的 `NavigationStack.append` push 模式,**没有暴露这两个 bug** —— 因为它有两层正交的保护:滚动位置存在 `MediaLibraryViewModel.scrollTop` + `NSScrollViewBridge` 桥到 AppKit;容器级 `@FocusState gridFocused` 在 openPreview 时置 false、closePreview 时置 true。消息流没有这两层。

### 根因诊断

两个 bug 同根 —— 都源于"预览走 NavigationStack push"这个架构选型对消息流不成立。

**Bug 1 根因**:`MessagesView` 用纯 `ScrollView { LazyVStack { ForEach } }`,没有任何滚动位置持久化。NavigationStack push 后 SwiftUI 重 layout LazyVStack,pop 回来时滚动位置在 macOS 14 上恢复不稳定 —— 这是 Apple 的已知未文档化行为(FB13322236 / FB13455901),社区有大量复现报告。

**Bug 2 根因**:`MessagesView` 没有容器级 `@FocusState`。`MediaDetailView` 关闭后焦点找不到明确归宿,被 AppKit 派发到 view tree 里下一个 focusable 元素 —— 通常是 toolbar 的搜索框/星标/刷新按钮,或 FilterSidebar 里的某个分类按钮。下一次空格按下激活了那个被偶然聚焦的元素,用户感受为"打开了别的媒体"。

`MessageMediaGrid5` 此前已经把媒体 cell 从 Button 换成 onTapGesture(line 65-72 注释解释了原因),堵住了"焦点跳回媒体卡再触发空格"这一具体路径 —— 这是正确的**局部**防御,但根本问题在容器层而不在 cell 层。

### 选型对比

研究了 6 个方案:

| 方案 | 否决理由 |
|---|---|
| ScrollViewReader + scrollTo(messageId) | LazyVStack 中未实例化的 id 调 scrollTo 不可靠;用户滚到中间打开预览,目标 id 通常已被 lazy 卸载 |
| `.scrollPosition(id:)` (iOS 17) | 对 LazyVStack 中未实例化 id 同样不稳定 |
| `ScrollPosition` 结构 (iOS 18) | macOS 15 才有,部署目标 macOS 14 排除 |
| `.fullScreenCover` | macOS 上**不存在该 API**;`.sheet` 默认 fitted 不全屏,`presentationDetents` 是 iOS-only |
| NSScrollViewBridge 改造消息流 | 变高 cell 不适合(NSScrollViewBridge 为等大方格虚拟化设计),改造代价大 |
| **ZStack overlay**(采纳) | view 不卸载 → 滚动位置自然保留;焦点链可控;与 native Photos/Finder/Mail 全屏预览模式一致 |

### 改动

只动一个文件:`MyNote/MyNote/MessagesView.swift`(±90 行)

1. **架构变更**:`.navigationDestination(for: MessagesPreviewDestination.self) { ... }` push 路由删除,改为 NavigationStack 内部 ZStack,预览作为 overlay 用 `if let state = previewState` 条件渲染。NavigationStack 保留空跑 —— ContentView 上提 toolbar 的路由契约要求每个 tab 自带 NavigationStack 以承接 navigationTitle / window title。
2. **新增状态**:
   - `@State previewState: PreviewState?` —— 单一来源,nil = 无预览。结构包含 `id: UUID` + `mediaList: [Media]` + `indexBox: IndexBox`(复用 `MediaPreviewDestination.swift` 已有的引用语义 box)。
   - `@FocusState feedFocused: Bool` —— 消息流容器焦点。
3. **删除状态**:`previewMediaList` / `previewIndexBox`(分散的旧 navigation 状态)、`MessagesPreviewDestination` struct 整个删除(不再需要 Hashable 路由)。
4. **消息流容器加焦点绑定**:在 `messagesScrollView` 末尾加 `.focusable(true) .focused($feedFocused) .focusEffectDisabled(true)`。
5. **改造 `openPreview`**:`feedFocused = false` 释放焦点 → 构造新 `PreviewState`(每次新 UUID)。删除 `navigationPath.append(dest)`。
6. **改造 `closePreview`**:`previewState = nil` → `DispatchQueue.main.async { feedFocused = true }` 一拍后恢复焦点。删除 `navigationPath.removeLast()`。
7. **MediaDetailView 加 `.id(state.id)`**:每次打开都是全新 view,防止连续打开不同消息的同一索引时 SwiftUI reuse,导致 AVPlayer / @FocusState / chromeVisible / mouseMonitor 等内部 state 残留。

### 设计要点

- **NavigationStack 不拆**:ContentView 的 toolbar 路由是 `switch selectedTab` 派发到各 tab 的 `static func ...Toolbar(...)`,机制本身没要求 NavigationStack 必须 push 什么;但 `.navigationTitle` 必须挂在 NavigationStack 内部 view 上才能提升到 NSWindow.title。空跑的 NavigationStack 成本是零,删它没好处。
- **`.id(state.id)` 不是为了 transition 动画**:transition 动画用 `.transition(.opacity)` + `.animation(value: previewState?.id)` 就够了。`.id(state.id)` 是为了**强制 SwiftUI 销毁旧 view 重建** —— 否则 MediaDetailView 内部的 `@State player: AVPlayer?` 在 view reuse 时不会通过 .onAppear 重建,即使 mediaList 换了也仍然播放旧媒体。
- **`DispatchQueue.main.async` 一拍恢复焦点**:同步在 closePreview 里 `feedFocused = true` 偶发被 SwiftUI 吞掉(view 拆除过程中 @FocusState 设置时机敏感)。async 一拍让焦点设置发生在 overlay 完全 dismiss、view tree 稳定之后,是社区验证过的稳妥写法,与 MediaLibraryView 此处的处理一致。
- **`IndexBox` 复用而非新建**:`MediaPreviewDestination.swift:13` 已有的引用语义 Int 包装,MediaLibraryView 也在用。`PreviewState` 持引用而非 `@State var previewIndex: Int`,是因为 MediaDetailView 翻页通过 `currentIndex` Binding 写回时,需要改的是 box 内的值而不是触发 SwiftUI 重建整个 previewState(后者会导致 view reuse 判定混乱,叠加 `.id` 之后甚至会重建 MediaDetailView)。
- **不动 MediaLibraryView**:它的 NavigationStack push 模式工作正常(焦点管理 + NSScrollViewBridge 已覆盖),没问题就不动,改它只是引入回归风险。
- **不动 MediaDetailView**:本次 bug 全部在 MessagesView 单文件可修。MediaDetailView 已有的 `onClose` 出口、内部 `@FocusState isFocused` 抢焦点、ESC/空格关闭、AVPlayer 重建、`PreviewTitle.shared` 窗口标题 —— 全部按现有方式工作,overlay 模式下的 onAppear/onDisappear 触发与 NavigationStack push/pop 等价。
- **不动 MessageMediaGrid5 的 onTapGesture 决策**:cell 不进入焦点链是**正交的局部防御**,与本次容器级修复互不冲突,保留即可。

### 验证清单

代码改完,等 Xcode 编译运行后手动验证:

- **Bug 1**:消息流向下滚 → 中间某条消息点开媒体 → 关闭 → 列表停在原位置(不回顶)。
- **Bug 2**:打开预览 → 空格关闭 → 再次空格 → 什么都不发生(不会打开别的媒体,toolbar/sidebar 按钮不被激活)。
- **预览翻页**:左右键切上一张/下一张,窗口标题跟着当前文件名变。
- **视频预览**:播放正常,关闭后无后台音频泄漏(`.id(state.id)` 保证 view 销毁 → onDisappear → player.pause + nil)。
- **多次打开**:连续打开三条不同消息的不同媒体 → 关闭 → 再打开,每次都正确显示对应媒体(验证 `previewState.id` UUID 隔离生效)。
- **切 tab**:消息 tab → 媒体 tab → 切回消息,滚动位置仍在(由 ContentView ZStack opacity 模式保证,本次不破坏)。
- **媒体库 tab**:NavigationStack push 模式未动,预览仍按原方式工作。
