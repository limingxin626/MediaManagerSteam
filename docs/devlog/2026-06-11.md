# 2026-06-11

## feat(backend): 半自动记账 —— 支付宝/微信账单导入

### 背景

用户想「自动记账」,主要消费在淘宝 / 京东 / 微信 / 支付宝。全自动抓取不可行(违反 App 条款、需存密码、有封号风险),故采用**半自动**:用户从支付宝/微信官方导出账单 → 放进 `slip/` 文件夹 → 跑 CLI 脚本自动解析、分类、去重、入库。

关键事实:淘宝/京东付款最终都经支付宝/微信结算,所以**只需解析这两份账单**即可覆盖绝大多数消费。淘宝/京东自身的「订单明细」不抓。

### 账单格式(基于两份真实账单实测,非猜测)

- **支付宝** `alipay_record_*.csv` —— **GBK** 编码,前若干行为说明,列头行含「交易号」,数据从列头下一行;17 列;尾部有「共N笔记录 / 导出时间」汇总行需跳过。关键列:金额(元)、收/支、交易状态、交易来源地、交易对方、商品名称、交易号(唯一)。
- **微信** `微信支付账单流水文件*.xlsx` —— **Excel(非 CSV)**,列头行含「交易时间」,数据从其下一行;11 列;`openpyxl` 返回的时间已是 datetime、金额为混合 int/float。关键列:交易时间、交易类型、交易对方、商品、收/支、金额(元)、当前状态、交易单号(唯一)。

为抵御账单改版(支付宝列名历史上改过多次),解析器一律**按列名定位**而非硬编码列序。

### 设计决策

- **收支都记**:income / expense / neutral(支付宝「不计收支」)用 `direction` 区分。
- **各源内去重**:`(source, biz_no)` 唯一约束保证重复导入幂等;不做跨源(支付宝×微信)去重。
- **过滤标记不丢弃**:退款 / 关闭 / 0元担保(等待确认收货)交易仍入库,但标 `excluded=1`,统计默认排除,保留审计痕迹。
- **分类**:关键词规则表(可增删)→ 字段信号兜底(支付宝来源地=淘宝→购物;微信交易类型转账/红包/扫码)→ 未命中归「未分类」。
- **入口**:CLI 扫 `slip/` 文件夹,本期不做 HTTP 接口 / 前端。
- **不入 SyncLog**:transaction 为服务端本地数据,Android 无需同步。

### 验证

- 解析条数对账单尾部声明 100% 吻合:支付宝 622 笔、微信 654 笔。
- 导入后总 1276 行(622+654);direction 分布 expense 958 / income 67 / neutral 251;excluded 171(支付宝140+微信31)。
- 0元担保交易(优衣库「等待确认收货」)正确 `excluded=1`。
- 分类抽查:优衣库→购物、蔬菜水果/熟食凉菜→餐饮,符合预期。

- **幂等**:二次运行 0 新增 / 1276 跳过,库内合计不变。
- **回归**:`message_fts*` 5 表 + 3 触发器完好;`from app import app` 正常,69 条路由。
