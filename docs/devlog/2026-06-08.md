# 2026-06-08

## fix(mac): 消息 tab 初始定位到最新 + 修复上拉加载触发不稳

### 背景

昨天的 commit d0c9fea 接入消息 tab,设计意图对齐 vue 端 `Message.vue`:
- 进入页面/切 filter 后落在最新消息(列表底部)
- 向上滚接近顶部第 10 条时 prefetch 拉更早的页

实际测试有两个 bug:

1. **初始位置在最上面(最旧)而非最下面(最新)**。结果用户无法直接向上滚加载更多 —— 必须先手动滚到底再回来才行。
2. **向上加载触发不稳定**。第一次能不能加载存疑(向下滚时可能路过 prefetch marker 已经被触发过),向上滚到顶要反复来回操作才能加载。

根因都在 `MessagesView.swift`,代码注释里写了几个对 SwiftUI 行为的假设,实测都不成立。

### 根因诊断

**Bug 1 —— `proxy.scrollTo("bottom-anchor")` 找不到目标:**
- `bottom-anchor` 是个 `Color.clear` 空 view,挂在 LazyVStack 数组之后。LazyVStack 首屏只渲染视口高度内的几条 + 一点 overscan,**末尾这个空 view 根本没被实例化** → `proxy.scrollTo("bottom-anchor")` 无目标 → 静默失败。
- 即使能找到,`MessageMediaGrid` 的 `LocalThumbView` 是异步加载,首屏布局完成后 cell 高度还在变化 → 一次性 scrollTo 即使生效也会被高度变化推走。
- `.onChange(of: messages.count)` 从 0 变 20 只触发一次,错过窗口就再也不滚。

**Bug 2 —— `ForEach(indices, id: \.self)` + prefetch marker 不会重建:**
- 原注释假设「不加 `.id()`,prepend 后旧 marker 销毁、新 marker 重建 → onAppear 重触发」—— 错的。
- ForEach 用 index (`\.self`) 作 id。Prepend 20 条后,`indices` 从 `0..<20` 变 `0..<40`,SwiftUI diff 看到 idx 0..19 的 view identity 不变,只是**内容**(MessageCard 绑定的 msg)被换。
- 嵌在闭包里的 `Color.clear` view identity 也不变 → onAppear 不重触发。
- 同时,"内容被替换" 还会引发**视觉错乱** —— 视口里原本的 MessageCard 被换成新插入的消息,用户感知到"内容在跳"。
- "滚到底再回来能触发" 是因为 LazyVStack 把上方 marker 真销毁回收(超出 overscan),再滚回来时是新建 view → onAppear 触发,纯粹侥幸。

### 改动

- `MyNote/MyNote/MessagesView.swift`(±38 行,只动一个文件)
  - `ForEach` 改用消息真实 id 作 diff key:`ForEach(Array(viewModel.messages.enumerated()), id: \.element.id)` —— prepend 时数组头部 20 条是**新建** cell,视觉上不抖;原 viewport 里的 cell view identity 不变,scrollTo anchor 恢复后用户视野无感。
  - prefetch marker 加 `.id("prefetch-marker-\(viewModel.messages.count)")` —— 每次 prepend 完成 count 变化,id 字符串变,SwiftUI 销毁旧 marker、构造新 marker → 新 marker 进入视口时 onAppear 重触发。这是把 vue 端 `rebindPrefetchTargets()` 的语义用 SwiftUI identity 系统实现。
  - 删除 `Color.clear.id("bottom-anchor")`(LazyVStack 末尾空 view,首屏没渲染没用)。
  - 新增 `scrollToLatest(proxy:)`:直接 `proxy.scrollTo(viewModel.messages.last!.id, anchor: .bottom)`,`ScrollViewProxy.scrollTo` 对未实例化 id 的契约是「强制 LazyVStack 实例化到目标可见」,不需要空锚点占位。0ms / 120ms 两次重压覆盖第一波 `LocalThumbView` 缓存 hit 完成后的 cell 高度跳变。
  - `handlePrefetchTrigger()` 加 `guard currentScrollY > 0`:首屏 / scrollToLatest 完成前,marker 可能就在视口里(尤其消息数少 prefetchIndex 落到数组末尾时),守门避免拉到「比首屏还旧」的页但用户根本没向上滚过。对齐 vue 端 `container.scrollTop > 0` 守门。
  - `.onChange(of: messages.count)` 的 scroll-to-bottom 分支改调 `scrollToLatest(proxy:)`,prepend 后视口恢复分支不动(保持原 `proxy.scrollTo(anchor, anchor: .top)`)。

### 设计要点

- **VM 接口零变动** —— `scrollToBottomPending` / `consumeScrollToBottomPending()` / `prependedPendingAnchor` / `consumePrependedPendingAnchor()` 协议保留;View 层只是消费方式变。
- **不引入 NSScrollViewBridge 重型方案**(MediaLibraryView 那种,代码复杂度高且只为 virtual grid 设计);保持原架构 `ScrollView` + `LazyVStack` + `ScrollViewReader`,只修正几个 SwiftUI 契约的误用。
- **两次重压代替三次**:Plan agent 原设计 0/120/360ms 三次,360ms 太长容易和用户初始操作(鼠标进场就开始滚)打架。两次足够覆盖图片缓存 hit 路径(~100ms),缓存 miss 的高度跳变用户已在底部看到最新条,后续向上看历史时自然滚动会修正,不必硬撑 scrollTo。
- **marker id 用 count 而非 index**:`"prefetch-marker-\(count)"` 比 `"prefetch-marker-\(prefetchIndex)"` 更稳 —— prefetchIndex 在 count ≥ 10 后恒为 10 不变,id 不变 view 不重建;count 每页 +20 一定变化。

### 已知行为

- **消息数 11-19 条且 hasMore=true 的过渡场景**:首屏 prefetchIndex 落到接近数组末尾,可能略滚一点就触发(scrollY 略大于 0)。无害 —— VM 内 `hasMore` / dedup 已防重复;再拉一页就脱离边界。
- **scrollToLatest 的 120ms 重压期间用户开始手动滚动**:第二次 scrollTo 会把用户拽回底部一次。窗口很短(单次跳变 60-120ms),实测可接受;如果反馈差再加 `lastProgrammaticScrollY` 检测被用户改过就跳过。
- **从 Preview 导航 pop 回 feed**:`messages.count` 未变化 → `onChange` 不触发 → 保持上次滚动位置(正确,无回归)。

### 契约保持

`MessagesViewModel` / `MessageRepository` / `MessageCard` / `MessageMediaGrid` 完全未碰。Backend / Vue / Android / Electron 不受影响。仅 `MessagesView.swift` 内部三处修正。
