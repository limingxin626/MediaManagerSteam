## feat(mac): 媒体网格新增 Finder 风格选中态(单击选中、双击预览、方向键移动、空格切换预览)

### 背景

Mac 端 `MediaLibraryView` 此前只有「单击直接打开预览」一种交互。Mac 用户对 Finder / 「照片」app 的两段式语义有强肌肉记忆 —— 单击只选中、再按空格或双击才打开预览、方向键在网格内移动选中 —— 缺失会显得这个 app「不够 Mac」。本次给网格补齐这套交互。

### 改动

- `MyNote/MyNote/MediaLibraryView.swift`
  - 新增 `@State var selectedIndex: Int?` + `@FocusState var gridFocused`,与原有 `showDetail` / `detailIndex` 并列(选中态与预览态解耦)
  - `MediaGridItem` 多接一个 `isSelected` 参数,选中时叠 `RoundedRectangle().stroke(Color.accentColor, lineWidth: 3)` overlay(不推 layout、不挡星标/视频徽标)
  - 单击改为只设 `selectedIndex`;双击同时设 selectedIndex 和打开预览(`onTapGesture(count:2)` 放前,SwiftUI 自动让双击优先,单击事件被延迟到 `doubleClickInterval` 后才触发)
  - 列数 `columnCount = 4` 提常量,`columns` 由 `Array(repeating:count:)` 生成,供方向键几何换算共用
  - `ScrollView` 包进 `ScrollViewReader`,ForEach cell 加显式 `.id(mediaItem.id)`;`selectedIndex` 变化时 `withAnimation(.easeOut(duration: 0.15)) { proxy.scrollTo(..., anchor: .center) }`,选中项自动滚到视口中央
  - 网格层加 `.focusable(true).focused($gridFocused).onKeyPress { ... }`,处理 4 个方向键 + 空格:方向键夹到 `[0, count-1]` 不 wrap;空格在预览未开时打开预览(选中为空时打开第 0 项),预览已开时返回 `.ignored` 让事件冒泡给 sheet
  - `.onAppear` 中 `gridFocused = true` 让网格首次出现即获得焦点
  - 默认选中第 0 项:`onChange(of: viewModel.media.count)` 处理 loadInitial / loadMore / filter 切换三种触发(`Media` 没有 Equatable,监听 count 已经能覆盖所有变化场景)
  - `onChange(of: viewModel.selectedMediaType)` / `showOnlyStarred` 把 selectedIndex 清掉,等数据回来再由 count 的 onChange 设回 0
  - `onChange(of: showDetail)`:预览关闭时 `selectedIndex = detailIndex`,把预览里最后看的那张同步回网格选中
- `MyNote/MyNote/MediaDetailView.swift`
  - `keyboardShortcuts` 里新增 `Button("Close on space") { dismiss() }.keyboardShortcut(.space, modifiers: [])`,与现有 ESC 关闭对称

### 设计要点

- 选中态放 View 本地 `@State`,不进 ViewModel —— 纯 UI 状态,跟数据加载/过滤/分页都不耦合,后续如要做「打开 app 恢复上次选中」再上提
- `onKeyPress` 是 macOS 14+ API,项目部署目标已是 14+(webp 解码要求),不引入新版本门槛
- 方向键不 wrap-around(末项 → 不跳回首项),与 Finder 一致
- 列数硬算几何足够 —— `columns` 是常量 4,等将来引入「可调缩略图大小」再把 `columnCount` 提成属性即可

### 已知行为

- 过滤按钮一旦被点击会抢走网格焦点(SwiftUI `Button` 的标准行为),用户再点一下网格区域即可恢复 —— 第一版不引入复杂焦点管理,观察后再加
- 单击高亮有 ~200ms 延迟(系统级 `doubleClickInterval`,Finder/Photos 一致),用户对这个延迟有预期

### 契约保持

`MediaDetailView` 的内部 ←/→/ESC 行为完全未动,新增的空格关闭只是与 ESC 对称的新增项,不影响既有调用方。`MediaLibraryViewModel` 完全未碰。Backend / Vue / Android / Electron 不受影响。

### 验证

## fix(mac): DateScrubber 日期跳转 / 拖动跟随 / 视觉边界全面修复

### 背景

5-29 给 Mac 端首次加了无限滚动 + DateScrubber 日期导航条,实测下来一组失败:

1. **日期跳转默认失败** — 拖到几年前的某天释放,主网格毫无反应。根因:`buckets` 仅按已加载到 `media[]` 的天分组(client-side groupBy),DateScrubber 拖到的目标日期通常落在「还没分页加载到的远期桶」,`buckets.firstIndex(where:)` 拿到 nil → `_scrollToBucketIndex` 不变 → UI 完全没反应。
2. **即便桶已加载到,跳转也常失效** — `ScrollViewReader.scrollTo("header-\(id)")` 在 `LazyVGrid` 里对**未实例化**的子节点 macOS 14 上经常静默无效。LazyVGrid 不预先报告未实例化子项的 frame,ScrollViewReader 算不出 anchor 位置。
3. **拖动期间圆点不跟手指** — `dragGesture.onChanged` 用 `if !wasDragging { onJump?(...) }` 守卫,只在拖动的**首帧**触发 `onJump`,后续帧全丢;`currentDate` 仅在 `onEnded` 写一次。
4. **maxDate 边界错** — `timelineMaxDate = viewModel.timeline.first?.date`(0 点),导致最新一天压在 0% 顶端,圆点和年标签全堆在轴顶。
5. **tooltip 被裁** — DateScrubber 整体 `.frame(width: barWidth + 34)` = 62,tooltip `.position(x: barWidth + 20, ...)` 超出框边界。
6. **年份标签密集时全部叠在一起** — 没有任何防重叠折叠(vue 端有 8% 折叠)。

整体根因是 mac 端没有对齐 vue 端 `useVirtualGrid` 已经在生产里跑通的「timeline 驱动桶布局 + 按桶懒加载 + 复合 cursor 边界判停」这一套模式。

### 改动

整体把 vue `useVirtualGrid` 模型平移到 mac,改动覆盖数据层、ViewModel、View 容器、DateScrubber 四层。

**数据层 — `MyNote/MyNote/MediaRepository.swift` + `MediaSource.swift`**

- 新增 `loadBucket(year:month:day:afterCursor:limit:type:starredOnly:)`,起始 cursor 走「次日 00:00:00 | Int.max」(对齐 vue `bucketStartCursor`),SQL 模板与 `list()` 共用,但遍历结果时遇到第一条 `created_at < 当天 00:00:00` 立即截断、标记 `isComplete = true`。tags 走原有 `fetchTags(db:mediaIds:)` 批量加载避免 N+1。返回 `(items, nextCursor, isComplete)`。
- `MediaSource` 协议加 `loadBucket`;`LocalMediaSource` 透传;`APIMediaSource` 暂 `fatalError`(只读 mac 第一阶段不走 API)。

**ViewModel — `MediaLibraryViewModel.swift` 整体重写**

- 删除原 `media: [Media]` 平铺数组,改为单一数据源 `buckets: [BucketLayout]` 由 `timeline: [TimelineEntry]` 派生。
- 新增 `BucketLayout`(key/year/month/day/count/rows/headerOffset/gridOffset/height/endOffset/date)、`BucketCacheEntry`(status/items/nextCursor)、`BucketStatus`(idle/loading/partial/complete/error)、`VisibleCell` 四个值类型。
- `rebuildBuckets()` 在 timeline + cellSize + cols 任一变化时重算几何;`totalContentHeight = buckets.last.endOffset` 是「整个媒体库的总滚动高度」,与已加载到的 items 数量无关 — 这是「跳到从未加载的远期桶」能命中的根本前提。
- `setContainerWidth(_:)` / `setViewportHeight(_:)` / `setScrollTop(_:)` 三个外部入口接 View 几何变化。
- `visibleBuckets` / `visibleCells` 派生属性按 `[scrollTop - PREFETCH, scrollTop + viewportH + PREFETCH]` 过滤,RENDER_OVERSCAN 更窄,只渲染必要 cell。
- 加载调度抄 vue:dwell 150ms 后入队、按到视口中心距离升序、并发上限 6;`loadBucketNow(_:)` urgent 插队;`setDispatchPaused(_:)` 拖动期间暂停。
- `scrollToDate(_:)`:`findBucketByDate` → 写 `jumpTargetY = bucket.headerOffset` + `jumpTrigger &+= 1` → View 的 NSScrollViewBridge 监听 jumpTrigger 推到 NSScrollView;同时 `loadBucketNow(key)` 抢占加载该桶。
- `findBucketByDate`:精确匹配 → 最近一个 `bucketDate <= date` → nil(timeline 为空)。
- `currentDate` 是 computed:取第一个 `endOffset > scrollTop` 的桶的 date,无则取末桶。
- `loadedFlatItems` computed `buckets.flatMap { bucketCache[$0.key]?.items ?? [] }`,供详情 sheet 和键盘选中拼接用,保持对 `MediaDetailView` 的 contract 不变。

常量与 vue 端对齐:`GAP=4`、`WINDOW_GAP=16`、`PAGE_LIMIT=100`、`PREFETCH_PX=800`、`RENDER_OVERSCAN_PX=400`、`DWELL_MS=150`、`MAX_CONCURRENT=6`、`HEADER_HEIGHT=32`、`TARGET_CELL=220`、`MIN_COLS=4`。

**View — `MediaLibraryView.swift` 网格容器重写 + 新增 `NSScrollViewBridge.swift`**

- 弃用 `LazyVGrid + ScrollViewReader + _scrollToBucketIndex` 的不可靠路径,改 `GeometryReader { ScrollView { ZStack(.topLeading) { 占位高 + 桶头 + 可见 cell } } }`,所有桶头和 cell 都通过 `.offset(x:y:)` 绝对定位 — 与 vue 端 `position:absolute` 同构。
- 新增 `NSScrollViewBridge: NSViewRepresentable`:`Coordinator` 在 `viewDidMoveToWindow` 时通过 `enclosingScrollView` 找到底层 `NSScrollView`,挂 `NSView.boundsDidChangeNotification` 监听把 `documentVisibleRect.origin.y` 回写到 binding;`updateNSView` 监听 `jumpTrigger` 变化时直接调 `clip.scroll(to: NSPoint)` 程序化跳转。这条路径**与子视图是否实例化无关**,跳到从未渲染过的桶也能精确命中。设了一次 `suppressFeedback` 标志避免 jump → boundsChange → setScrollTop 回环。
- 桶头每个都 `.frame(width: geo.size.width, alignment: .leading)`,避免 HStack + Spacer 在 ZStack 下塌成 Text 固有宽度。
- 未加载到的 cell 渲染灰色占位 `Rectangle().fill(Color.gray.opacity(0.12))`,不会因为缺数据让 layout 跳动。
- `GeometryReader.onAppear` / `onChange(of: geo.size)` 把宽高传给 ViewModel,负责重算 cellSize/cols 并触发 dispatch。

**DateScrubber — `DateScrubber.swift` 整体重写**

- `currentDate` 由 `@Binding` 改为 `let currentDate: Date`(单向数据流);拖动期间走本地 `@State draggingDate: Date?` 覆盖显示;`onEnded` 后延迟 50ms 清空,让外部 currentDate 接管。
- `dragGesture.onChanged` 每帧都更新 `tooltipDate` / `tooltipY` / `draggingDate`;`onJump` 通过 `scheduleJump` 节流(`pendingJumpDate` + 单飞 `Task @MainActor` + `Task.yield`)每帧最多触发一次。
- `onEnded` 触发一次 `onJumpFinal`,**不再依赖 onChanged 首帧守卫**。
- tooltip 浮层移出右轨道 ZStack,作为外层 `ZStack(.topLeading)` 顶部子项,通过 `.position(x: -32, y: tooltipY)` 显示在 scrubber 左侧外部,不受 totalWidth 限制。
- 年份标签防重叠:`yearLabels` 按 `top` 升序遍历,与已 kept 的最后一个 `top` 差 < 8% 的直接丢掉;长刻度本身全部保留,只折叠文字。逻辑直接对齐 vue `DateScrubber.vue` 的 `yearLabels` computed。
- `timelineMaxDate` 在 `MediaLibraryView` 计算时改为最新一天的 `23:59:59`,minDate 仍是最早天 00:00:00 — 避免最新一天的圆点贴在 0% 顶端。
- `onContinuousHover` 在非拖动状态下更新 tooltip 位置,鼠标离开 tooltip 消失。

**`MediaLibraryView` 集成 scrubber**:`onJump` 闭包先 `setDispatchPaused(true)` 暂停可视范围调度、再 `scrollToDate(date)`;`onJumpFinal` 闭包 `scrollToDate(date)` + `setDispatchPaused(false)`,落定桶通过 `scrollToDate` 内部的 `loadBucketNow` 已经触发加载。

### 设计要点

- **timeline 是 mac 端的「全量索引」**,先于任何 media items 加载完成。BucketLayout 单条 ~80 字节,3650 天也只 ~290KB,完全可接受。
- **跳转走 NSScrollViewBridge 而非 ScrollViewReader**,根本上规避了 SwiftUI ScrollViewReader 对未实例化节点失效的问题。
- **dispatch 暂停 + 拖动节流**:拖动经过 N 个桶时只对落定桶 `loadBucketNow`,中间桶被 setDispatchPaused 守住不触发加载;松手后才恢复 dwell 调度。
- **拖动单向覆盖 currentDate**:外部 `currentDate` 是从 `scrollTop` 派生的「权威值」,但拖动期间用户手指位置才是权威,所以本地用 `draggingDate` 临时覆盖,释放后 50ms 延迟清空让外部接管 — 比双向 `@Binding` 不会出现「拖到一半外部回写覆盖」的抖动。

### 契约保持

- 后端 `/media/timeline` 和 `/media` 端点完全未触碰,backend / vue / android / electron 不受影响。
- 现有 spec `media-day-buckets` / `media-timeline` 描述的是 vue 端契约,本次新增 `mac-media-timeline-nav` 平行 spec,不修改任何已有 requirement。
- `MediaDetailView` 对外接口完全未动(`mediaList: [Media]` + `currentIndex: Binding<Int>` + `hasMore: Bool` + `onNeedMore: () async -> Void`),只是这里 hasMore 暂传 false(详情翻到末尾不再触发自动加载下一页) — **临时降级**,后续若需要可在详情末尾边缘触发 `loadBucketNow(下一个 bucket.key)`。

### 已知风险 / 待 Mac 上验证

1. **tooltip 用 `.position(x: -32, y:)`** — 显示在 scrubber 左外侧 32pt。如果 ScrollView 或父层有 `clipped()`,会被裁掉。Mac 上看到第一眼就能判断,若被裁可改成 `.popover` 实现。

## fix(mac): VirtualScrollView 自建 NSScrollView,根治「只显示首页 + scrubber 不跟手」

### 背景

上一轮的 `NSScrollViewBridge` 用了「SwiftUI `ScrollView { ZStack { ... } }.background(NSScrollViewBridge())`」的桥接思路 —— bridge 内部丢一个空 `NSView`,在 `viewDidMoveToWindow` 时沿 `superview` 链找 `enclosingScrollView`,再挂 `boundsDidChange` 监听把 `scrollTop` 回写到 ViewModel。

在 Mac 上跑后暴露两个症状:

1. **首屏之外全是空白**:鼠标滚轮可以滚动屏幕,但越过首屏后 cell 全部消失,只剩占位灰色;跳到任何远期日期也是空白
2. **DateScrubber 圆点不跟主网格滚动**

两个症状同一根因 —— **bridge 没真正挂上 `NSScrollView`**:
- `boundsDidChange` 永远没回调 → `viewModel.scrollTop` 始终 `0`
- `visibleBuckets` / `visibleCells` 只算第 0 像素那一屏
- `currentDate` 由 `scrollTop` 派生,也不动

具体原因有两条:
- SwiftUI 的 `.background()` 把 NSView 挂的位置依不同 macOS 版本而异,有时根本不在 `NSScrollView` 的子树里,`superview` 链断在 `_NSHostingView`/`_NSScrollViewWrapper`,`enclosingScrollView` 拿不到
- 即便挂到了,SwiftUI `ScrollView` 包了一层壳,`NSClipView.boundsDidChangeNotification` 不一定会按预期触发(SwiftUI 自管滚动状态,不一定走 AppKit 通知)

### 改动

- `MyNote/MyNote/NSScrollViewBridge.swift` 整文件重写
  - 抛弃「找 enclosingScrollView」的桥接思路
  - 新结构体 `VirtualScrollView<Content: View>: NSViewRepresentable`,在 `makeNSView` 里**自己 `new` 一个 `NSScrollView`**,`documentView` 用自定义 `FlippedHostingView<AnyView>`(`isFlipped = true`,坐标原点左上)包 SwiftUI 内容
  - `Coordinator` 持有 `weak scrollView: NSScrollView?`,在 `makeNSView` 里直接挂 `NSView.boundsDidChangeNotification` + `frameDidChangeNotification` 到 `sv.contentView` —— 引用一直在手里,通知 100% 收得到
  - 桥反测三个值回写给 ViewModel:`@Binding scrollTop`(boundsDidChange)、`viewportHeight` / `containerWidth`(frameDidChange + 首次 async)
  - 程序化跳转:`updateNSView` 比对 `jumpTrigger` 变化时直接调 `sv.contentView.scroll(to:)` + `sv.reflectScrolledClipView()`,设 `suppressFeedback = true` 吞掉紧随的 boundsChange 回写一次,防 jump → boundsChange → setScrollTop 抖动
  - `updateNSView` 每次都重设 `host.rootView = AnyView(content())`,让 SwiftUI ForEach 在 ViewModel `@Published` 变化时拿到最新的 `visibleBuckets`/`visibleCells` 重新绘制
  - `Coordinator` **不标 `@MainActor`**(`NSNotification` selector 触发线程不可控),内部用 `DispatchQueue.main.async` 写回 `@Binding`
- `MyNote/MyNote/MediaLibraryView.swift`
  - `grid` 私有计算属性整段重写:删掉 `GeometryReader { ScrollView { ZStack {...} } .background(NSScrollViewBridge) .onAppear/.onChange(of: geo.size) }`
  - 改为直接 `VirtualScrollView(contentHeight: ..., scrollTop: ..., viewportHeight: ..., containerWidth: ..., jumpTrigger: ..., targetY: ...) { ZStack(alignment: .topLeading) { 桶头 + cells } }`
  - 桶头宽度 / ZStack 显式 frame 用 `viewModel.containerWidth`(由桥反测后回写),不再依赖 `GeometryReader { geo.size.width }`
- `MyNote/MyNote/MediaLibraryViewModel.swift`
  - 新增 `@Published var containerWidth: CGFloat = 0`(给桥反向 binding 用)
  - `setContainerWidth(_:)` 头部加 `containerWidth = width`,保持其他派生逻辑不变

### 设计要点

- **自己 new `NSScrollView`** 而不是找 SwiftUI 的:消除「桥挂没挂上」这个不可控变量。所有依赖滚动状态的功能(可视范围派生、程序化跳转、scrubber 反向跟随)都建立在一个我们持有的 `NSScrollView` 引用上,任何时刻都可以打断点确认是否生效
- **`FlippedHostingView`**:`NSView` 默认底为 (0,0),flipped 后变为左上,与 SwiftUI `.offset(x:y:)` 的 top-left origin 语义一致;否则 cell 的 y 坐标会从底往上叠,虚拟化布局全部错位
- **`containerWidth` 由桥反测,不再走 `GeometryReader`**:SwiftUI 嵌在 NSViewRepresentable 内部时 GeometryReader 测出的宽度是 hosting view 本身,不是 NSScrollView clipView 宽度,差一根滚动条就错位;直接读 `sv.contentView.bounds.width` 是源头数据,准
- **`updateNSView` 每次都重设 `host.rootView`**:SwiftUI 父级触发更新时,这是把新值传进内部 SwiftUI 树的唯一路径(NSHostingView 不会自动响应外部 `@Published`,因为它隔了 NSViewRepresentable 边界)
- **`suppressFeedback` 一次性吞回写**:程序化跳转后必然触发一次 boundsChange,如果回写到 `scrollTop` 再触发 `dispatchFetches` 会产生不必要的调度;吞一次让链路在跳转后稳定
- **`Coordinator` 不标 `@MainActor`**:`NSNotification` 投递线程在 AppKit 内部不可控,标 `@MainActor` 会让 selector 签名必须 `nonisolated` + `Task @MainActor in` 跨界,反而容易触发 Swift 6 isolation 错误;统一在 selector 内 `DispatchQueue.main.async` 是最稳的写法

### 已知风险 / 后续

- `host.rootView = AnyView(content())` 每次 update 都执行,内部 SwiftUI 树会失去 `@State` —— 当前传入的内容是纯派生(ZStack + ForEach over ViewModel),无本地状态,所以安全。如果未来在 content 闭包里加 `@State`,需要改成「diff 后再赋值」或者把状态上提到 ViewModel
- 首屏短暂空白(< 1 frame):`makeNSView` 后 `viewportHeight` 由 `DispatchQueue.main.async` 才设,在那之前 `visibleBuckets` 用 0 算 → 算出空集合;下一轮 update 立即恢复,肉眼不可见
- 桥目前只支持纵向滚动 + 单 documentView,不支持嵌套 ScrollView / 横向滚动 —— 本场景不需要
