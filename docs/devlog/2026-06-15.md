## refactor(全栈): 统一 backend / Mac / Vue 三端 path/URL 字段语义

### 背景

repository 改造后 `media.file_path` 已经是「相对 repo 根的 forward-slash 路径」(2026-06-06 那次迁移),但 schema 出口的派生字段三端语义对不上,问题出现得很隐蔽:

- backend `MediaUrlMixin._fill_urls` 给 `thumb_path` 填的是**绝对**本地路径(`E:/.../data/thumbs/123.webp`),给 `file_path` 直接透传 DB 的**相对**路径。
- Vue 在 `vue/src/utils/media.ts` 里 `resolveMediaUrl` 优先读 `file_path` 当**绝对**路径用(Electron `file://`)—— 迁移后 `file_path` 是相对的,这是个静默 bug,只是 Vue 现在很少跑详情没暴露。
- Mac `MediaRecord.toUIModel` 自己拼 `fileUrl = "/data/{filePath}"`,但 backend `url_for` 的真正约定是 `/{repo_id}/{file_path}` —— 也错。
- 同一字段名,三端心智模型对不上 = 后续每加一个端就要踩一次坑。

Android 那边没有这问题(它本来就只走相对 URL + sync baseUrl 拼接),但顺手钉死了 URL 字段语义。

### 字段标准

| 字段              | 语义                                                | 谁消费              |
|-------------------|-----------------------------------------------------|---------------------|
| `repo_id`         | repository id                                       | 全员                |
| `file_path`       | 相对 repo 根, forward-slash(DB 权威)              | Mac 自己拼          |
| `local_file_path` | 绝对本地路径(`repo_root + file_path`)              | Vue / Electron      |
| `local_thumb_path`| 绝对本地路径 `{DATA_ROOT}/data/thumbs/{id}.webp`    | Vue / Electron      |
| `file_url`        | **相对** URL `/{repo_id}/{file_path}`               | Android(拼 baseUrl) |
| `thumb_url`       | **相对** URL `/data/thumbs/{id}.webp`               | Android(拼 baseUrl) |

URL 字段选**相对**而非绝对,理由:Android sync 时 OkHttp/Retrofit 的 `baseUrl` 是它说话的那台 backend,相对 URL 通过 `HttpUrl.resolve(baseUrl, relativeUrl)` 拼,跟「这次 sync 来自哪台 host」永远一致。

绝对 URL 的坑(全部规避):
1. 多 host 缓存污染(在公司 `192.168.x` sync,回家换 `10.0.x` 缓存就指错了)
2. `Request.base_url` 在反代/容器/`0.0.0.0` 监听下不可靠
3. 没有 `PUBLIC_BASE_URL` env 这种「搬家就要改」的运维负担

- `app/schemas/base.py` —— `MediaUrlMixin` 重写:字段从 `(file_path, file_url, thumb_path, thumb_url)` 改成 `(repo_id, file_path, local_file_path, local_thumb_path, file_url, thumb_url)`,删 `thumb_path`(绝对本地路径)字段名 —— 它的语义被拆到 `local_thumb_path`。

## refactor(全栈): DATA_ROOT 子目录展平,去掉多余的 `data/` 层

### 背景

历史上 `get_thumbs_dir() / get_preview_dir() / get_actor_cover_dir()` 把目录拼成 `{DATA_ROOT}/data/{thumbs,preview,actor_cover}/`,但 `/data` URL prefix 通过 StaticFiles 挂到 `DATA_ROOT` 根 —— 这意味着 HTTP URL `/data/thumbs/123.webp` 物理对应 `DATA_ROOT/thumbs/123.webp`,而实际文件被写到 `DATA_ROOT/data/thumbs/123.webp`,**URL 段和物理路径长期错位**。Mac 端 GRDB 直读绕开 HTTP,Vue Electron `file://` 直读绝对路径,两端都不走 HTTP 路由所以错位没暴露;Android 走 HTTP 但缩略图 fallback 路径比较少触发。

这次顺手把布局拍平 —— `{DATA_ROOT}/{thumbs,preview,actor_cover}/`,URL 段保持 `/data/thumbs/...` 不变(因为 `/data` 还是挂 DATA_ROOT 根),结果**URL 和物理路径真正对齐**了。

### 改动

- `backend/app/config.py` —— `get_thumbs_dir / get_preview_dir / get_actor_cover_dir` 去掉中间的 `"data"` segment。`/data` URL mount 不动。
- `MyNote/MyNote/Models.swift` —— `Media.localThumbURL / localPreviewURL`、`MessageMediaItem.localThumbURL`、`Actor.localAvatarURL` 都去掉 `.appendingPathComponent("data", ...)` 那一层。
- `MyNote/MyNote/MessageCard.swift` —— `loadAvatar()` 拼 url 同上。
- `MyNote/MyNote/Settings.swift` / `AnimatedVideoView.swift` / `MAC_TODO.md` —— 注释里的旧路径同步。
- `CLAUDE.md` / `README.md` / `openspec/specs/mac-message-data/spec.md` —— 文档同步。

### 物理文件迁移

代码不带自动迁移。用户在更新前需要手动:

```bash
cd $DATA_ROOT
mv data/thumbs ./thumbs
mv data/preview ./preview        # 可能不存在
mv data/actor_cover ./actor_cover
rmdir data                       # 空目录可删
```

不迁的话:Mac 网格立即变空白(读 `DATA_ROOT/thumbs/` 不到)、Vue Electron 同样、smart_tag_service 等也读不到 thumb。

### 不做

- 不写自动迁移代码(一次性操作,加 startup 检测代码反而是债)。
- 不改 URL 段(`/data/thumbs/...` 沿用,Android 收到的 thumb_url / 旧缓存都不受影响)。
- 不删 `/data` URL prefix(/data → DATA_ROOT 这个 mount 还有用,数据库以外的子目录都通过它出 HTTP)。

### 验证

1. 手动 `mv` 完成后启动 backend,`HEAD http://127.0.0.1:8002/data/thumbs/<某个id>.webp` 返回 200(之前因为 URL/物理错位实际 404,只是没人留意)。
2. Mac Xcode Build → Run,网格缩略图正常加载。
3. `grep -r "data/thumbs\|data/preview\|data/actor_cover" --include='*.swift' --include='*.py' MyNote backend` 应只剩 backend `/data/thumbs` URL 字面量(get_thumbnail_url / smart_tag_service)。

## feat(全栈): 记账只读浏览页 `/transactions`

### 背景

2026-06-11 已落地半自动账单导入(CLI + service + 模型),但只能跑命令行看打印。这次补**只读的网页前端**,把数据库里 1276 笔流水真正变成「可看」的 —— 月份切换 + 分类堆叠条 + 流水列表 + 分类/支出收入/含排除过滤。这一档刻意不做导入、不做规则 CRUD,见 [2026-06-11 后续 TODO](2026-06-11.md);先把「看」做对,再决定要不要把上传/规则搬到网页。

### 改动

#### Backend(新增只读 router)

- `app/schemas/transaction.py`(新建)—— `TransactionOut` / `TransactionListResponse` / `MonthlySummary` / `MonthBucket` / `CategorySlot`,纯 read DTO。
- `app/routers/transactions.py`(新建)—— `prefix=/transactions`,4 个 GET:
  - `GET /transactions` —— 游标分页 + `year/month/category/direction/excluded` 过滤。游标格式 `"{txn_time_iso}|{id}"`,按 `(txn_time DESC, id DESC)` 排序;同一秒多条流水靠 id 兜底,延续 media 的复合游标范式。`limit + 1` 检测 `has_more`。
  - `GET /transactions/summary/monthly?year=&month=&direction=expense` —— 透传 `transaction_service.monthly_summary`。
  - `GET /transactions/months` —— `strftime('%Y-%m', txn_time)` 分组,返回所有有数据的月份 + 笔数 + 计入支出合计(给前端月份下拉用)。计入支出用 `case((excluded==0)&(direction=='expense'), amount, else_=0)` 而非 SQLite 专有 `iif`。
  - `GET /transactions/categories` —— 已出现的 category,按使用频次降序。
- `app/routers/__init__.py` —— 把 `transactions_router` 加入 `all_routers`。

#### Frontend(新增视图 + 导航)

- `vue/src/views/Transactions.vue`(新建)—— 顶部月份下拉(选项 = `/transactions/months` 返回);月度汇总卡(总额 + 水平堆叠条 + 分类明细表,纯 CSS 不引图表库);过滤栏(支出/收入/全部 + 分类下拉 + 含已排除 checkbox);流水列表(`useInfiniteScroll`,limit=50)。复用 `useApi` / `useToast` / `formatDateTime`。
- `vue/src/router/index.ts` —— 新增 `/transactions` 路由,懒加载。
- `vue/src/components/Navbar.vue` / `BottomNavBar.vue` —— 左侧 + 底部导航各加一项「记账」(¥ 图标,Heroicons currency-dollar)。

### 设计决策

- **只读**:页面不含上传/编辑/规则增删。`Transaction` 表已存在的字段就足够展示 —— 改 category 或新增规则仍走 CLI / DB。
- **过滤默认排除「已排除」**:`excluded=0` 是默认 query,与统计口径一致;勾选 checkbox 才显示退款/0元担保记录(视觉上还会 line-through + 灰字 + 「不计统计」标签)。
- **月份是主轴**:列表强制按月窗口拉取(无「全部时间」),与汇总卡同步;一年/全部聚合是另一种交互,本期不做。
- **不引图表库**:堆叠条 + 分类列表用 div + 百分比宽度 + 固定调色板(10 色循环),够看。引 echarts/d3 是另一档体量。
- **复合游标 vs 简单游标**:支付宝单文件 600+ 流水里,同一秒多笔的情况实际存在(扫码连付/批量退款),所以走复合游标。
- **categories endpoint 与规则表无关**:返回的是 `transaction.category` 实际出现过的值(包含 `未分类`),不读 `txn_category_rule`;意图是「让用户筛已有数据」而非「展示规则全集」。

### 不做

- 不做账单上传 / 网页端导入(CLI 仍是唯一入口)。
- 不做规则 CRUD(改 `txn_category_rule` 仍需直接操作 DB)。
- 不做行内改 category(本期纯只读)。
- 不引 echarts/d3。
- 不入 SyncLog(transaction 本来就不同步)。

### 验证

1. `python api.py` 启动,`from app.routers import all_routers` 路由数从 11 → 12;`/openapi.json` 含 4 个 `/transactions*` 路径。
2. `vue-tsc --noEmit` 通过(EXIT=0)。
3. 手动:`pnpm dev` 打开 http://localhost:5173/transactions ——
   - 月份下拉选项与 `GET /transactions/months` 一致。
   - 默认选最近一个月,汇总卡显示总额、堆叠条按分类比例划分、分类明细按金额降序。
   - 切月份 → 汇总卡和流水列表同步刷新。
   - 「含已排除」勾选 → 出现 `excluded=1` 行,金额变灰 + 划线 + 标「不计统计」。
   - 滚到底自动加载下一页(limit=50)。
   - 左侧 + 底部导航出现「记账」入口,激活态正确。

### 追加: 流水列表支持排序(时间 / 金额)

- 后端 `app/routers/transactions.py`:
  - 新增 query `sort: 'time'|'amount'`(默认 time)和 `order: 'asc'|'desc'`(默认 desc)。
  - 游标格式从写死的 `"{txn_time}|{id}"` 改成通用 `"{primary}|{id}"` —— primary 由 sort 决定(ISO 时间字符串 / `str(amount)` 数字字符串)。`_format_primary` 和 `_parse_primary` 一对函数承接编解码。
  - WHERE 子句和 ORDER BY 都按 `(sort_col, id)` 复合,id 兜底同值稳定性(amount 同金额的两笔流水不会跳过)。`order=asc` 时 WHERE 用 `>`,`order=desc` 时用 `<`,跟 ORDER BY 方向匹配。
  - 排序 module type alias `SortField` / `SortOrder` 显式 Literal,FastAPI 自动生成 enum 校验。
- 前端 `vue/src/views/Transactions.vue`:
  - 过滤栏右侧新增「排序」按钮组(时间 / 金额),激活态显示 ↑/↓ 箭头。
  - `toggleSort(field)`:同字段再点切换方向,换字段重置为该字段默认方向(时间默认 desc,金额默认 desc —— 用户最常看「最大支出」)。
  - sort/order 加入 `useInfiniteScroll` 的 fetchFn,改动触发 `resetList()` 重头拉。

#### 为什么不只做时间一种?

支付宝/微信账单一个月平均 100+ 笔,「我这个月最大那笔花了什么」是真实场景 —— 时间倒序找不到,得手动翻几屏。加金额排序覆盖这个高频需求即可,日期范围 / 多字段排序留到有人提为止。

## chore(backend): 账单全量重标 + 转为人工(LLM)月度打标流程

### 背景

`transaction_service.DEFAULT_RULES`(关键词规则表)在 1276 条真实流水上跑出来的分类太不准了 —— 比如"会员"50 分把 Steam 充值之外杂项也吃掉、"二维码→餐饮"过于粗暴、缺少"医疗/娱乐/学习/服装/住房"等常见类别。决定**整体推倒**:旧规则废弃,新流程改成 LLM 配合用户逐条打标。

### 流程

1. `scripts/bills_split_by_month.py` —— 解析 `slip/` 下 CSV+xlsx,按月切分到 `scratch/bills_by_month/YYYY-MM.json`,每条加空 `category` 字段。
2. `scripts/bills_print_month.py YYYY-MM` —— 紧凑打印一个月的待标记录(Claude 读起来省 token)。
3. Claude 读列表 → 产出 `scratch/mappings/YYYY-MM.py`(`biz_no → category` 字典),遇到判断不准的就当面问用户。
4. `scripts/bills_apply_categories.py` —— stdin 收 mapping,按 biz_no 匹配写回到月份文件。未命中报错。
5. `scripts/bills_inject_labeled.py [--apply]` —— 清空 `transaction` + `txn_category_rule`,按所有月份 JSON 注入。dry-run 默认。

### 类目体系(14 类)

餐饮 / 交通 / 购物 / 服装 / 数字订阅 / 生活缴费 / 医疗 / 娱乐 / 学习 / 旅游住宿 / 人情往来 / 转账理财 / 住房 / 其他

完整定义 + 判断原则 + 易混淆场景:`docs/bills-categories.md`。

### 设计决策

- **JSON 是真相之源,DB 只是物化**:`scratch/bills_by_month/*.json` 必须保留;`inject` 任何时候可以重跑,DB 重建幂等。这样改一条 category 就是改 JSON + 重 inject,不用 SQL。
- **不再自动分类**:`txn_category_rule` 表清空,`transaction_service.classify()` 留着但只在老 `import_bills.py` 用 —— 而老 CLI 不再走主流程。新月度账单的归类一律走 LLM。这是用 LLM 的判断换关键词规则的低准确率。
- **AA / 红包 / 转账的边界靠用户**:同样是"个人转账 5000",可能是房租水电(人情往来)、给家人(人情往来)、车票代付(交通)、礼金(人情往来)、房屋定金(住房);商户名一致但语义完全不同。这种条目 Claude 不猜,挂在月度对话里问用户,记入 `bills-categories.md` 的"必问"清单。
- **excluded(退款/0元担保)按原消费类目归**:消费 + 退款两条都标同一类,统计自动跳过 —— 不要全推到"其他",否则按 category 看月度时 excluded 行会污染"其他"。
- **`scratch/mappings/*.py` 保留**:不是临时产物。下次新月份打标可以参考往期对类似商户的判断,避免反复抛同样问题。

## feat(全栈): 记账页支持月份范围筛选

### 背景

只读记账页(本文上一节)落地后只能按单月看。实际使用里「近 3 个月支出趋势」「整年某个类目总和」这种需求很常见 —— 一个一个月切看不到累计。这次把月份维度从「单选」升到「起止范围(闭区间,以月为单位)」,汇总卡按整个范围聚合,流水列表也按范围拉。

### 改动

#### Backend

- `app/routers/transactions.py`:
  - 新增 `_month_window(y, m)` 和 `_resolve_range(...)` 两个 helper。`_resolve_range` 优先吃 `from_year/from_month/to_year/to_month` 四元组,起止反了自动 swap;否则回落到 `year/month` 单月;都没传就不过滤。
  - `GET /transactions` 新增 4 个 query:`from_year / from_month / to_year / to_month`,旧的 `year / month` 仍然支持(只有 Vue 在调,但保留是几行代码的事)。
  - 新增 `GET /transactions/summary/range?from_year=&from_month=&to_year=&to_month=&direction=expense`,返回 `RangeSummary`。
- `app/services/transaction_service.py`:新增 `range_summary(db, fy, fm, ty, tm, direction)`,复用 `_summary_query`,内部同样会 swap 起止。
- `app/schemas/transaction.py`:新增 `RangeSummary`(from_year/from_month/to_year/to_month/total/by_category)。

#### Frontend(Vue)

- `vue/src/views/Transactions.vue`:
  - 单 `selectedYM` → `fromYM` + `toYM` 两个下拉,同源 `months`,默认起止都填最近一个月。
  - `normalizedRange` computed 把两个下拉规范化成 `{from, to}` 闭区间(swap + 一端缺失退化成另一端单月),`fetchSummary` / 列表 fetchFn 都从它取参。
  - 汇总卡标题改成 `summaryTitle` computed:单月显示 `YYYY-MM`,跨月显示 `YYYY-MM ~ YYYY-MM`,后面附 `· N 个月`。
  - 汇总改打 `/transactions/summary/range`(不再用 `/summary/monthly`)。
  - `watch(selectedYM)` → `watch([fromYM, toYM])`,onMounted 初值同时赋两个(Vue 批一次 tick → 只触发一次 fetch)。

### 设计决策

- **闭区间 + 月粒度**:`from=2026-01 & to=2026-06` 含 6 个月,SQL 用 `[from_start, to_next_month_start)` 半开区间。「以月为单位」是用户的原话,不上 daypicker 复杂度,直接复用 `/transactions/months` 已有的下拉数据。
- **起止反了自动 swap,不报错**:用户拖错下拉是常见操作,swap 比弹错误对话框更顺手。前后端各 swap 一次(前端 swap 之后才发请求,后端再 swap 一次纯粹是兜底)。
- **保留 monthly endpoint**:`/summary/monthly` 没删,虽然 Vue 不再调,但它是个独立合约,后续如果有别的地方按单月聚合(dashboard 之类),不用再造一遍。删除属于「为不存在的清理而清理」,留下成本是零。
- **不引日期范围 picker 组件**:两个 select 已经够直观,装一个 daterange-picker 还要适配 v-calendar 主题、对齐暗色模式,代价/收益不划算。

### 不做

- 不改流水列表按月分组展示(目前是平铺时间倒序;跨月时用户看到的是连续流水,符合预期)。
- 不存范围状态到 localStorage / URL(刷新就回默认最近月,够用;后续要做再加)。
- 不做「最近 N 个月」「本年」预设按钮(可加但不是这次需求,保持最小)。

### 验证

1. `python -c "from app.routers.transactions import router; print([r.path for r in router.routes])"` 返回 5 条,包含新 `/transactions/summary/range`。
2. `vue-tsc --noEmit` 通过(EXIT=0)。
3. 手动:打开 `/transactions` ——
   - 起止默认填最近一个月,汇总标题显示单月 `YYYY-MM`。
   - 起选 `2026-01`、止选 `2026-06`,汇总标题变 `2026-01 ~ 2026-06 · 6 个月`,总额 = 6 个月支出和,堆叠条/分类明细按整段范围比例分布。
   - 故意起选大、止选小,前端会自动 swap,结果一致。
   - 切起/止任一边,列表立即重头拉(按选定范围 + 当前过滤 + 排序)。

## feat(全栈): 记账页流水行内编辑

### 背景

记账只读页 + 月份范围都跑通后,真实使用里第二个高频痛点是「这一行分类标错了 / 这一行其实应该排除 / 这家商户名是乱码想改成可读」—— 之前只能 SQL 直改 + 重启,门槛太高。这次给每行加内联编辑,4 个用户语义字段可改:`category` / `excluded` / `counterparty` / `product`,其它(amount/txn_time/source/biz_no/raw_*)是原始记录性质,刻意不开放。

### 改动

#### Backend

- `app/schemas/transaction.py`:新增 `TransactionUpdate`,所有字段都是 `Optional`,`extra="forbid"` 拒绝未知字段。
- `app/routers/transactions.py`:新增 `PATCH /transactions/{txn_id}`,`exclude_unset=True` 只写显式提供的字段,404 / 400(无字段)分别报错。直接 `db.commit()`,无 SyncLog(transaction 本来就不进同步)。

#### Frontend(Vue)

- `vue/src/views/Transactions.vue`:
  - 每行右侧加铅笔按钮(SVG inline,激活时换成上箭头表示「收起」)。
  - 点击 → 该行下方展开内嵌表单(grid-cols-2):counterparty / product 文本框、category select(复用 `categories.value`,带「(无)」选项)、不计入统计 checkbox、保存 / 取消。
  - 保存时做 diff:只把跟原值不同的字段塞进 payload,完全没改就直接关闭。文本字段 `trim() === ''` 落 null,category 空串同样落 null。
  - 调 `api.patch<Txn>(...)`,响应直接覆盖 `items.value[idx]`。改了 category / excluded 顺手 `fetchSummary()` 刷新汇总卡(总额、堆叠条会变)。

### 设计决策

- **只开放 4 个字段**:用户语义字段(怎么显示/怎么归类)可改,原始记录字段(amount/txn_time/source/biz_no/raw_*)不开放 —— 改这些等于伪造账单,真要改应该是上游导入逻辑的事。`extra="forbid"` 后端兜底。
- **diff 提交,不全字段覆盖**:节省日志噪声、避免误把别人后续改的字段回写覆盖。`exclude_unset=True` 配合前端 diff,语义干净。
- **保存后只 patch 单行 + 刷汇总**:不重头拉列表(代价大且会跳走滚动位置)。但如果改了 category,排序为「金额」+ 当前 category filter 等场景下,这一行可能本不该在当前列表里 —— 接受这个轻微的「需要刷一下才完全一致」,换更顺手的体验。
- **不写回 scratch JSON**:JSON 是历史标注真相、DB 是物化产物 —— 但增量手改只动 DB 是用户在 UI 里直接表达的意思,下次跑 `bills_inject_labeled.py --apply` 才会被 JSON 覆盖。这个边界在 `docs/bills-categories.md` 已经说明,inject 是「重新物化」语义。
- **行内展开 vs Modal**:展开更直接(不打断滚动 / 不挡其它行),复杂度也低。Modal 留给「真编辑全字段」场景,这里 4 个字段两列 grid 完全够。
- **铅笔按钮颜色**:默认 `text-gray-400`,hover 才变 primary,跟来源徽章 + 金额字号统一,不喧宾夺主。

### 不做

- 不开放 amount/txn_time/source/biz_no 编辑(语义错位,见上)。
- 不做删除单条(误删风险 > 收益;DB 直删即可)。
- 不做批量编辑(后续需要再加;现在没有批量选中机制)。
- 不写回 scratch JSON / 不重跑 inject(同上)。
- 不加 keybinding(Esc 取消 / Cmd+Enter 保存):后续按需,行内编辑通常鼠标流。

### 验证

1. `python -c "from app.routers.transactions import router; print([(r.path, list(r.methods)) for r in router.routes])"` 含 `('/transactions/{txn_id}', ['PATCH'])`。
2. `vue-tsc --noEmit` 通过(EXIT=0)。
3. 手动:打开 `/transactions` ——
   - 每行尾部出现铅笔按钮,点击 → 行下方展开编辑面板。
   - 改 category(下拉)+ 勾选「不计入统计」+ 改 counterparty 文本 → 保存 → 行立即更新,汇总卡总额 / 堆叠条 / 分类明细同步刷新。
   - 不改任何东西点保存 → 直接关闭,无请求。
   - 取消 → 关闭,数据不变。
   - 后端用 curl 试 `PATCH /transactions/1 -d '{"foo":1}'` → 422(extra forbid)。
