## refactor(全栈): 统一 backend / Mac / Vue 三端 path/URL 字段语义

### 背景

repository 改造后 `media.file_path` 已经是「相对 repo 根的 forward-slash 路径」(2026-06-06 那次迁移),但 schema 出口的派生字段三端语义对不上,问题出现得很隐蔽:

- backend `MediaUrlMixin._fill_urls` 给 `thumb_path` 填的是**绝对**本地路径(`E:/.../data/thumbs/123.webp`),给 `file_path` 直接透传 DB 的**相对**路径。
- Vue 在 `vue/src/utils/media.ts` 里 `resolveMediaUrl` 优先读 `file_path` 当**绝对**路径用(Electron `file://`)—— 迁移后 `file_path` 是相对的,这是个静默 bug,只是 Vue 现在很少跑详情没暴露。
- Mac `MediaRecord.toUIModel` 自己拼 `fileUrl = "/data/{filePath}"`,但 backend `url_for` 的真正约定是 `/{repo_id}/{file_path}` —— 也错。
- 同一字段名,三端心智模型对不上 = 后续每加一个端就要踩一次坑。

Android 那边没有这问题(它本来就只走相对 URL + sync baseUrl 拼接),但顺手钉死了 URL 字段语义。

### 字段标准

| 字段              | 语义                                                | 谁消费              |
|-------------------|-----------------------------------------------------|---------------------|
| `repo_id`         | repository id                                       | 全员                |
| `file_path`       | 相对 repo 根, forward-slash(DB 权威)              | Mac 自己拼          |
| `local_file_path` | 绝对本地路径(`repo_root + file_path`)              | Vue / Electron      |
| `local_thumb_path`| 绝对本地路径 `{DATA_ROOT}/data/thumbs/{id}.webp`    | Vue / Electron      |
| `file_url`        | **相对** URL `/{repo_id}/{file_path}`               | Android(拼 baseUrl) |
| `thumb_url`       | **相对** URL `/data/thumbs/{id}.webp`               | Android(拼 baseUrl) |

URL 字段选**相对**而非绝对,理由:Android sync 时 OkHttp/Retrofit 的 `baseUrl` 是它说话的那台 backend,相对 URL 通过 `HttpUrl.resolve(baseUrl, relativeUrl)` 拼,跟「这次 sync 来自哪台 host」永远一致。

绝对 URL 的坑(全部规避):
1. 多 host 缓存污染(在公司 `192.168.x` sync,回家换 `10.0.x` 缓存就指错了)
2. `Request.base_url` 在反代/容器/`0.0.0.0` 监听下不可靠
3. 没有 `PUBLIC_BASE_URL` env 这种「搬家就要改」的运维负担

### 改动

#### Backend

- `app/schemas/base.py` —— `MediaUrlMixin` 重写:字段从 `(file_path, file_url, thumb_path, thumb_url)` 改成 `(repo_id, file_path, local_file_path, local_thumb_path, file_url, thumb_url)`,删 `thumb_path`(绝对本地路径)字段名 —— 它的语义被拆到 `local_thumb_path`。
- `app/routers/sync.py` —— `_message_snapshot` 内联的 media 字典 + `_media_snapshot` 函数,手动加 `local_file_path / local_thumb_path`,删 `thumb_path`(原 schema 也没这字段,这里只清掉过期注释)。
- `app/schemas/media.py` —— `VideoPreviewItem` docstring 更新。

#### Vue

- `vue/src/utils/media.ts` —— `resolveThumb` / `resolveMediaUrl` 改读 `local_*_path`(优先)+ `*_url`(HTTP 兜底)。删掉对 `file_path` / `thumb_path` 这两个字段的所有引用 —— 之前那个「file_path 优先」就是迁移后的隐藏 bug。
- `vue/src/types.ts` —— `Media` / `MessageMediaItem` / `VideoPreviewItem` 类型扩展 `local_file_path` / `local_thumb_path`(OpenAPI 还没重生成,手扩)。
- `vue/src/views/Media.vue` / `vue/src/components/dashboard/RecentMedia.vue` —— `mapToPreviewItem` / `openPreview` / `openSmartPreview` 透传 `local_*_path` + 删 `thumb_path` 引用。
- `vue/src/components/MediaPreview.vue` —— `openFileLocation` 改用 `local_file_path` 给 Electron `showItemInFolder`(不存在时提示);`onFileSelected` 替换后同步 `local_*_path` 字段;`getMediaUrl` 改用 `local_file_path` 而非 `item.file_path`(后者现在是相对路径,直接当 URL 用会炸)。
- `vue/src/views/MediaDetail.vue` —— `openMediaFolder` 同样改用 `local_file_path` + 友好提示。
- `vue/src/views/Actor.vue` / `vue/src/views/Message.vue` / `vue/src/views/Media.vue` —— `handleMediaRotated` 的 cache-bust 从 `file_path` 改成 `local_file_path`(只有后者是 Electron 实际加载的 URL,加 `?t=` 才有意义)。

#### Mac

- `MyNote/MyNote/MediaRecord.swift` —— `toUIModel` 不再拼 `fileUrl / thumbPath / thumbUrl`(之前还拼错了)。Mac UI 全走 `localFileURL` / `localThumbURL` 计算属性。
- `MyNote/MyNote/Models.swift` —— `Media` / `MessageMediaItem` 删 `fileUrl / thumbPath / thumbUrl` 三个 String 字段和对应 `CodingKeys`;`localFileURL` 删除 `legacyExtractedURL` 兜底,直接走 `RepositoryManager.resolve(...)`;两个 `legacyExtractedURL` 私有函数整段删。
- `MyNote/MyNote/MessageMediaGrid.swift` —— `asMedia()` 删掉对三个废弃字段的引用。

#### 文档

- `CLAUDE.md` —— 第 69 行那条「auto-fill URL fields」短句改成完整的 6 字段语义表 + 相对 URL 设计理由。

### 不做

- 不改 DB schema / 不写 migration(`media.repo_id` + `media.file_path` 已经是目标语义)。
- 不改 backend `config.py`(`url_for / resolve_to_absolute / get_thumbnail_path` 直接复用)。
- 不引入 `PUBLIC_BASE_URL` env / `server_url` 绝对 URL 字段(相对 URL 让客户端 sync baseUrl 作唯一真相源)。
- 不重新生成 `api.generated.ts`(后续 OpenAPI 重生时会一并对齐,这次先手扩)。

## refactor(全栈): DATA_ROOT 子目录展平,去掉多余的 `data/` 层

### 背景

历史上 `get_thumbs_dir() / get_preview_dir() / get_actor_cover_dir()` 把目录拼成 `{DATA_ROOT}/data/{thumbs,preview,actor_cover}/`,但 `/data` URL prefix 通过 StaticFiles 挂到 `DATA_ROOT` 根 —— 这意味着 HTTP URL `/data/thumbs/123.webp` 物理对应 `DATA_ROOT/thumbs/123.webp`,而实际文件被写到 `DATA_ROOT/data/thumbs/123.webp`,**URL 段和物理路径长期错位**。Mac 端 GRDB 直读绕开 HTTP,Vue Electron `file://` 直读绝对路径,两端都不走 HTTP 路由所以错位没暴露;Android 走 HTTP 但缩略图 fallback 路径比较少触发。

这次顺手把布局拍平 —— `{DATA_ROOT}/{thumbs,preview,actor_cover}/`,URL 段保持 `/data/thumbs/...` 不变(因为 `/data` 还是挂 DATA_ROOT 根),结果**URL 和物理路径真正对齐**了。

### 改动

- `backend/app/config.py` —— `get_thumbs_dir / get_preview_dir / get_actor_cover_dir` 去掉中间的 `"data"` segment。`/data` URL mount 不动。
- `MyNote/MyNote/Models.swift` —— `Media.localThumbURL / localPreviewURL`、`MessageMediaItem.localThumbURL`、`Actor.localAvatarURL` 都去掉 `.appendingPathComponent("data", ...)` 那一层。
- `MyNote/MyNote/MessageCard.swift` —— `loadAvatar()` 拼 url 同上。
- `MyNote/MyNote/Settings.swift` / `AnimatedVideoView.swift` / `MAC_TODO.md` —— 注释里的旧路径同步。
- `CLAUDE.md` / `README.md` / `openspec/specs/mac-message-data/spec.md` —— 文档同步。

### 物理文件迁移

代码不带自动迁移。用户在更新前需要手动:

```bash
cd $DATA_ROOT
mv data/thumbs ./thumbs
mv data/preview ./preview        # 可能不存在
mv data/actor_cover ./actor_cover
rmdir data                       # 空目录可删
```

不迁的话:Mac 网格立即变空白(读 `DATA_ROOT/thumbs/` 不到)、Vue Electron 同样、smart_tag_service 等也读不到 thumb。

### 不做

- 不写自动迁移代码(一次性操作,加 startup 检测代码反而是债)。
- 不改 URL 段(`/data/thumbs/...` 沿用,Android 收到的 thumb_url / 旧缓存都不受影响)。
- 不删 `/data` URL prefix(/data → DATA_ROOT 这个 mount 还有用,数据库以外的子目录都通过它出 HTTP)。

### 验证

1. 手动 `mv` 完成后启动 backend,`HEAD http://127.0.0.1:8002/data/thumbs/<某个id>.webp` 返回 200(之前因为 URL/物理错位实际 404,只是没人留意)。
2. Mac Xcode Build → Run,网格缩略图正常加载。
3. `grep -r "data/thumbs\|data/preview\|data/actor_cover" --include='*.swift' --include='*.py' MyNote backend` 应只剩 backend `/data/thumbs` URL 字面量(get_thumbnail_url / smart_tag_service)。
