# 账单分类规则

本文档约定支付宝/微信账单导入后的人工(LLM)打标分类体系。
旧的自动关键词规则(`txn_category_rule` 表 + `transaction_service.DEFAULT_RULES`)已废弃 —— 太不准。

新流程:**每月底人工(LLM)对当月新增账单逐条打 category,然后清月份注入。**

## 类别清单(14 类)

| 类别 | 涵盖 | 反例/边界 |
|---|---|---|
| **餐饮** | 外卖、餐厅、咖啡奶茶、便利店、超市、扫码个人小额(默认餐饮) | — |
| **交通** | 打车、地铁公交、火车票/机票、加油、停车、高速、车险、年检、共享单车单次骑行(非月卡)、地铁卡/一卡通充值 | 哈啰单车**月卡**走数字订阅 |
| **购物** | 网购实物(淘宝/京东/拼多多)、日用百货、家电、运费、理发/造型、运动营养品 | 服装走"服装";药店走"医疗";房屋家装走"住房" |
| **服装** | 衣鞋袜帽配饰、奥莱、优衣库/UR/马登/拉夫劳伦/迪卡侬 等 | 运动 T 恤(Gymshark)也归服装,不是购物 |
| **数字订阅** | Steam/Valve、App Store、视频会员、AI 订阅(MiniMax/Cursor)、网盘、哈啰单车月卡、高德地图省钱卡、剪映 svip、网易云会员、小米影视、雷神加速器、PIKCLOUD、京东虚拟服务、CodePlanPlus | 哈啰单车按次骑行 → 交通 |
| **生活缴费** | 水电燃气、话费(联通/移动)、网费、流量包、物业费 | 房租走"住房" |
| **医疗** | 药店、医院、挂号、理疗按摩(医疗目的)、护视(隐形眼镜) | 健身按摩、SPA 走"娱乐";装饰镜框走"购物" |
| **娱乐** | 电影、景点门票(白瀑寺/颐和园/植物园/鸡鸣寺/牛首山/海坨山谷/苍源/大美西山)、游乐(金属狗/可乐手作 DIY)、健身按摩(星之健身/POIN SPA)、游戏机周边、淘票票 | 学习类讲座/会务费走"学习" |
| **学习** | 书、课程、培训、孔阳国学堂(讲座/会务/实践活动) | — |
| **旅游住宿** | 酒店、民宿、Airbnb、携程订房 | 机票/火车票走"交通";门票走"娱乐";押金 X 退走"其他" |
| **人情往来** | 红包(发/收)、给个人转账(私人往来)、礼金、退还/AA 之外的转账;微信群发的春节红包(收发都算) | AA 餐费走"餐饮";帮买票之类委托代付按真实用途归类 |
| **转账理财** | 基金买入(蚂蚁/纳斯达克)、花呗自动还款、信用卡还款、纯资金搬运 | 实物消费哪怕是 neutral 也不算 |
| **住房** | 房租、房屋定金、装修材料(墙纸/家具) | 物业、水电、网费走"生活缴费" |
| **其他** | 罚款、捐赠、出入境证件费、打印复印、烟花爆竹、押金类 excluded(已退) 单笔、忘了是什么的扫码 | 能识别的尽量归 13 类 |

## 关键判断原则(从 2025-06 ~ 2026-06 实测中沉淀)

### 默认假设
- **扫码个人小额(10-100 元)** 一律 **餐饮**,除非商户名明显是其他(发型店/快印店/烟酒店)
- **微信"商户消费"+ 商户名带 餐饮/烧烤/熟食/小吃/水果/蔬菜/超市/便利** → **餐饮**
- **高速被扫支付 / 停车场 / ETCP / 一码通行 / 一点停 / 知位停车** → **交通**
- **辰域智控** 水/电/燃气表 → **生活缴费**

### 资金搬运 vs 真实消费
- `direction=neutral` 的基金买入、花呗还款 → **转账理财**
- `direction=expense` 但实物已退(`excluded=1`)的退款单 → 按原消费类目归;两条(消费+退款)都标同类,统计会自动跳过
- 微信红包发出/收到 **不论金额** → **人情往来**
- 群收款 `excluded=0` 的 AA 餐 → **餐饮**

### 商户名歧义 → 必问用户
以下场景不要自己猜,挂在月度对话里问用户:
- 扫码个人**大额**(≥150 元) —— 可能是 AA/借款/还款
- 商户名仅含人名 + 金额异常(比如某人 264 元、某人 800 元)
- 转账备注空白 + 金额 ≥500 元 —— 可能是房租定金/借款/礼金
- 商户名是公司名但看不出业务(写"商贸"/"科技"/"服务"等)

## 月度运行流程

每月底(或新季度初)拿到新一份支付宝 CSV + 微信 xlsx 后:

```bash
# 1. 把账单文件丢到 slip/ 下
# 2. 切分到月份 JSON(脚本会扫整个 slip/,所以已存在的月份会被重新生成 —— 需要先备份已标好的 JSON 或只跑新增月份)
cd backend
python scripts/bills_split_by_month.py

# 3. 跟 Claude 一起逐月打标:
#    a) python scripts/bills_print_month.py 2026-07     # 紧凑列出当月
#    b) Claude 读列表,产出 scratch/mappings/2026-07.py(biz_no → category 字典)
#    c) python scratch/mappings/2026-07.py | python scripts/bills_apply_categories.py
#       会报告未命中的 biz_no(打错单号时抛错)

# 4. 全部月份标完后,清库 + 注入
python scripts/bills_inject_labeled.py            # dry-run,只打印
python scripts/bills_inject_labeled.py --apply    # 真做
```

**注意:**
- `bills_inject_labeled.py --apply` 会清空 `transaction` **和** `txn_category_rule` 全表,重新基于 `scratch/bills_by_month/*.json` 注入。**所以历史月份 JSON 必须一直保留** —— 它是真相之源,DB 只是它的物化。
- apply 前自动备份:`cp Data/db.sqlite3 Data/db.sqlite3.bak.YYYYMMDD`
- `import_bills.py`(老的 CLI)还在,但跑完后会有大量"未分类",所以**别用**;新流程一律走 split → mappings → inject。
