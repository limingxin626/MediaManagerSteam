## 同步策略 3 个 Gap 修复

### 背景

审计当前 Android↔Backend 同步策略，识别出 3 个真实数据丢失/漏拉路径，本次一并修掉。其余「过度设计」（LWW 比较、SSE、retry x10 等）作为单用户 LAN 暂不动。

### Gap #2：Tag 关联表删除未写 SyncLog

`backend/app/services/sync_log_service.py` 之前只监听 Message/Actor/Media/Tag 实体本身的 flush，移除某条 message 上的 tag 关联只改 `message_tag` 表，不会触发宿主 message 的 dirty，Android 下次拉到的 message 快照仍带旧 tag。

**修法**：注册 `event.listen(Message.tags, 'append'/'remove', _bump_updated_at)` 与 `Media.tags` 同款。关联集合变化时把宿主 `updated_at` 推到当前时间，自然进入 `session.dirty`，被现有 `_after_flush` 写为 MESSAGE/MEDIA UPSERT 日志。覆盖 `message.py` 内所有 `db_message.tags = …`、`other.tags = []`、`sync_tags_from_text(merge=False)` 等路径，无需逐处侵入。

### Gap #3：同页去重吞掉 DELETE

`backend/app/routers/sync.py` 去重逻辑用 dict 后写覆盖；同 entity 在一页内若先 DELETE 后 UPSERT（极端：删除后立刻同 id 新建），DELETE 会丢，客户端误以为实体仍在。

**修法**：维护 `deleted_keys` 集合，看到 DELETE 即标记，后续同 key UPSERT 不再覆盖最终 op。

### Gap #1：Android 不发 since_id

后端 `/sync/changes` 已支持 `(since, since_id)` 复合游标，客户端只发 `since`，相同毫秒多条变更可能漏拉；同时 `syncIncremental` 翻页时把 `cursor` 直接重置为 `body.server_time` 也错误地跳过了同毫秒未拉的条目。

**修法**：

1. `SyncService.getChanges` 增加 `sinceId: Long = 0L` 入参。
2. `RemoteChangesResponse` 增加 `next_cursor_id: Long?`。
3. `SyncResult.Success` 增加 `serverCursorId`。
4. `MessageRepository.syncIncremental(since, sinceId)`：翻页时 `has_more=true` 用 `next_cursor`/`next_cursor_id` 续，最后一页才用 `server_time` 作为下次起点。
5. `SyncWorker`：新增 `KEY_LAST_SYNC_ID` 持久化第二维。

- `android/app/src/main/java/com/example/myapplication/data/service/SyncService.kt`

## 同步策略瘦身：移除单用户场景下的过度设计

### 背景

修完 3 个 gap 后，进一步把多设备/多用户假设带来的机器拆掉。本次 4 个改动均为减法，无新增能力。

### 1. 删除 `/sync/events` SSE 端点

`backend/app/routers/sync.py` 移除 `sync_events()` 路由；`backend/app/services/sync_log_service.py` 删除 `_subscribers` / `subscribe` / `unsubscribe` / `_broadcast` / `_after_commit` 全部 SSE 相关代码 (~70 行)。Android 端从未订阅，每次 commit 仍走全队列广播是纯浪费。实时刷新继续靠 `SyncWorker.scheduleImmediateSync` 在网络恢复时触发一次拉取。

### 2. 撤掉 LWW 时间戳比较

`sync.py` 的 `_upsert_message` / `_upsert_actor` / `_upsert_media` 三处删除 `incoming_updated <= existing.updated_at` 守卫，无条件覆盖。单端时钟漂移不再静默丢推送。`updated_at` 仍作为版本戳写入，方便人工排查。

### 3. Outbox 重试 10 → 3 次

`SyncOutboxRepository.MAX_RETRY_COUNT` 改为 3。LAN 下网络要么通要么后端没起，10 次只会拖延错误暴露；失败后 15 分钟周期任务会自然重试。

### 4. SyncLog 保留期 90 → 365 天

`SYNC_LOG_RETENTION_DAYS = 365`，错误文案同步从「90天」改为「1年」。SyncLog 表不大，单设备很少离线 90 天以上，但放宽到 1 年几乎零成本。

### 修改文件

- `backend/app/routers/sync.py`
- `backend/app/services/sync_log_service.py`
- `android/app/src/main/java/com/example/myapplication/data/repository/SyncOutboxRepository.kt`

### 验证

- `./gradlew compileDebugKotlin`：通过。
- backend Python AST 语法检查：通过。
- 待人工跑：启动 backend 应不再注册 SSE 端点；POST `/api/sync/apply` 不再因为 `updated_at` 比较被拒。

## 离线消息同步状态可见性修复

### 背景

用户反馈：安卓端离线时新建消息，UI 先显示「同步中」、几秒后徽标消失；重启 App 也不再显示，根本无法分辨这条消息究竟有没有同步成功。

排查根因：
1. `MessageViewModel` 的 `isOnline` 仅基于 `NetworkMonitor.isWifiConnected`，但服务器是局域网自托管，**走出家门连了别的 WiFi**或服务器宕机时，仍会被误判成在线 → 进 try → 失败后静默降为 `PENDING_SYNC`。
2. `MessageCard` 的 `when (sendStatus)` 只画了 `PUSHING` 与 `PUSH_FAILED` 两个分支，`PENDING_SYNC` 落入 `else`，与 `SYNCED` 共享展示，徽标消失。
3. 重启 App 不会自动重推 `MSG_STATUS_PENDING_SYNC` 的消息：WorkManager 只重试 `SyncOutboxItem` 队列，消息走的是独立的 `create-from-client` 通路。

### 改动

**1. 后端新增 `GET /health` 探活端点**（`backend/app/routers/health.py` + `routers/__init__.py`）：返回 `{"status":"ok"}`，不查 DB、不鉴权，毫秒级响应。

**2. `NetworkMonitor` 扩展为后端可达性检查**：构造函数注入 `SyncPreferences`，新增 `isBackendReachable`（30s 周期 + 网络变化触发 `GET /health`，3s 超时）和组合 `isOnline = !isOfflineMode && isBackendReachable`。`onLost` 时立刻置 false，避免延迟。

**3. `DatabaseManager`** 调整初始化顺序，先建 `syncPreferences` 再传给 `NetworkMonitor`。

**4. `MessageCard`** 新增 `MSG_STATUS_PENDING_SYNC` UI 分支：橙色（`tertiary`）小圆点 + 「待同步 · 重试」，与 `PUSHING`（旋转 spinner）/ `PUSH_FAILED`（红色）三态视觉清晰区分。

**5. `MessageViewModel`**：
   - `isOnline` 改用 `networkMonitor.isOnline.value`，离线时直接置 `PENDING_SYNC`（之前依赖入库初值，但初值在离线分支下没有正确设过）。
   - `sendMessage` / `retrySync` 的 `catch` 拆分：`IOException` → `PENDING_SYNC`（可恢复）；其他异常（HTTP 4xx/5xx、序列化错误）→ `PUSH_FAILED`（用户需手动介入）。不再「为了不吓用户」一律降级为待同步。
   - 新增 `retryAllPending()`：扫描所有 `PENDING_SYNC` 消息复用 `retrySync` 幂等重推（后端按客户端 id 去重）。

**6. `MessageDao` / `MessageRepository`** 新增 `getMessagesBySendStatus`。

**7. 触发点接入**：
   - `MainActivity.onCreate`：监听器从 `isWifiConnected` 改为 `isOnline` 边沿，false→true 时同时触发 `SyncWorker.scheduleImmediateSync` + `messageViewModel.retryAllPending()`。
   - `MessageListScreen` 的 `LaunchedEffect(Unit)` 进入页面时调一次 `viewModel.retryAllPending()`，覆盖「冷启动且后端已可达」场景。

## 媒体预览收藏按钮状态不刷新

### 背景

用户反馈：进入 `MediaViewerScreen` 后点击右上角收藏按钮，图标不变；退出再进则正确显示新状态。说明 DB 已写入，但当前页 UI 未重组。

### 根因

`MediaViewerContent` 的收藏点击回调先 `launch` 协程调 `toggleMediaStarred`（挂起 + DB IO + outbox 推送），完成后再回到主线程改 `mutableMediaList`。在用户感知的瞬时窗口里，本地 `SnapshotStateList` 没被更新，按钮自然不变。同时 `mutableMediaList = mediaList as? MutableList<Media>` 对 `List<Media>` 形参的 `as?` 在 stable 参数 + 跳过重组的组合下不够稳，星标读取也依赖 `currentMedia.starred` 的快照订阅，链路过长。

### 修法

`MediaViewerScreen.kt:384-407` 重写收藏按钮：

1. 把 `idx`/`starred` 提到 `IconButton` 之外，先**乐观更新** `mutableMediaList[idx]`，再 `launch` 异步落库；UI 立即反映。
2. 校验 `idx in indices && it[idx].id == targetId`，避免快速翻页时改错条目。
3. Icon/contentDescription/tint 全部读 `starred` 局部变量，订阅路径最短。

## 消息列表保留滚动位置（Telegram 风格）

### 背景

每次进入 Message 页面都从最新（视觉底部）开始，浏览历史时切到其他 Tab / 进预览再回来，位置全丢，体验和 Telegram 群组聊天差距大。

### 根因

`MessageListScreen.kt:96` 的 `rememberLazyListState()` 仅活在当前 composition 周期，离开页面即销毁。`reverseLayout=true` 让 index 0 变成视觉底部，所以重建时默认就回到「最新」。也没有任何持久化机制。

### 改动

**1. `SyncPreferences.kt` 新增 3 个方法**：`getMessageScrollAnchor()` / `setMessageScrollAnchor(messageId, offsetPx)` / `clearMessageScrollAnchor()`。键存在已有的 `sync_preferences` SharedPreferences 里，跨进程持久化。

**2. `MessageListScreen.kt` 三段新逻辑**：

- **首次恢复**：`LaunchedEffect(pagingItems.itemCount)` 等 Paging 加载到第一批后，按持久化的 `messageId` 在已加载窗口里 `peek` 查 index，命中就 `scrollToItem(index, offset)`。未命中且总数 < 500 时等下一次扩页再试，超过仍找不到（多半被删了）就放弃，停在底部。
- **持久化写入**：`snapshotFlow { firstVisibleItemIndex to firstVisibleItemScrollOffset }` 配 `debounce(400)` + `distinctUntilChanged`，把当前首可见项对应的 `message.id` 与 offset 写回 prefs。`DisposableEffect.onDispose` 兜底再写一次。
- **新消息「在底部才跟随」**：`isAtBottom = firstVisibleItemIndex == 0 && offset < 80`；`snapshotFlow { itemCount }.drop(1)` 监听新增，仅当处于底部时才 `animateScrollToItem(0)`，否则不打断历史浏览。

为什么按 **message id** 而不是 index：`reverseLayout` + Paging prepend + 同步 prepend 都会让 index 漂移，id 才是稳定锚点（也已经是 LazyColumn 的 `key`，见 `MessageListScreen.kt:195`）。

发送自己的消息后强制跳底部的逻辑（`MessageListScreen.kt:268`）保留，符合直觉。
