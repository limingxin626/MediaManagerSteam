用户输入: $ARGUMENTS

## 确定日期

- 今天记为 `TODAY`（当天，用于日历查询）
- 回顾起始日记为 `YESTERDAY`（用于回顾邮件/Teams/文件）
- 用户输入包含具体日期 → 将该日期作为 `YESTERDAY`，次日作为 `TODAY`
- 用户输入为空 → 默认 `TODAY` = 当天
- **周一特殊处理**：如果 `TODAY` 是周一，`YESTERDAY` = 上周五（即回顾周五、周六、周日三天的数据）
- 非周一时，`YESTERDAY` = 昨天

## 数据获取

采用两级渐进策略，按需深入：

1. **聚合查询** — 并行查询：
   ```
   # 今天的日程（最重要，优先获取）
   data_search(source="calendar", start_date="TODAY", end_date="TODAY")
   # 本周剩余日程预览（TOMORROW 到本周五；周五则省略）
   data_search(source="calendar", start_date="TOMORROW", end_date="FRIDAY")
   # 回顾期的通讯和文件动态（周一时覆盖周五~周日，其他时候仅昨天）
   data_search(source="emails", start_date="YESTERDAY", end_date="昨天的实际日期")
   data_search(source="teams", start_date="YESTERDAY", end_date="昨天的实际日期")
   data_search(source="files", start_date="YESTERDAY", end_date="昨天的实际日期")
   ```
   返回 summary 统计 + 每条的 Subject/From/Priority/ID + 按优先级裁剪的正文预览。
   这一步提供全局视图，足以完成大部分分析。

2. **深入查看** — 对需要了解完整内容的条目，用 ID 批量拉取：
   ```
   # 今日会议的完整详情（参会人、议程/正文）
   data_open(source="calendar", ids=[会议类事件的ID列表])
   # 会议类事件 = IsOnlineMeeting=true 或 AttendeeCount>2
   # 简单事件（≤2人、全天事件）用 search 的预览即可，不需要 data_open

   data_open(source="emails", ids=[需要深入的邮件ID])
   data_open(source="teams", ids=[需要深入的消息ID])
   ```
   大字段会温和截断（邮件正文≤4000字符，Teams消息≤20条/正文≤500字符），适合大多数场景。

## 生成简报

分析获取到的数据，生成简报：

1. 写入临时文件 `reports/briefing.md`（固定文件名，每次覆盖）
2. 调用 `write_message_from_md(file_path="reports/briefing.md", tag_list=["简报"])` 将简报内容写入笔记系统

### 简报模板

```markdown
# 晨间简报 YYYY-MM-DD（周X）

> 一句话总结今天的工作节奏（例如："今天有3个会议，上午较密集，下午有整块时间处理邮件"）

## 今日日程
按时间顺序列出今天的所有会议和事件。对会议类事件（通过 data_open 获取了完整详情的），
展示参会人、地点和议程要点；对简单事件仅列基本信息。标注[需准备]的会议。

| 时间 | 会议 | 参会人 | 备注 |
|------|------|--------|------|
| HH:MM-HH:MM | 会议名称 | 张三、李四、王五 +3人 | 地点；议程要点/[需准备] |

今天没有会议时显示"今天没有会议安排，可以专注处理待办事项。"

## 本周预览
本周剩余天数的日程概览，按天分组。仅列时间和主题，不展开详情。
周五或无后续事件时省略此板块。

**周三 (MM-DD)**
- HH:MM-HH:MM 会议名称
- HH:MM-HH:MM 会议名称

**周四 (MM-DD)**
- HH:MM-HH:MM 会议名称

## 昨日回顾 / 周末回顾
2-3 句话总结回顾期内最重要的事，帮助用户快速回忆。
周一时标题改为"周末回顾"，覆盖周五至周日的内容。

## 谁找过我
昨天的 Teams 消息 + 未读邮件按发件人聚合，注明主要内容和是否需要回复。
- **人名**（渠道）— 内容摘要 [需回复] / [已处理] / [仅通知]

## 待处理事项
从昨天的邮件和消息中提取需要行动的事项，按优先级排列：
- 【来源：发件人】事项描述

## 今日建议
基于昨天的未处理事项和今天的日程，给出 2-3 条今天应该优先做什么的建议。

## 文件动态
OneDrive/SharePoint 中昨天有变动的文件，注明文件名、修改者、所属项目。

## 可延后处理
不紧急的邮件/通知，空闲时浏览即可。
```

### 写作规则

- 全程中文
- 语气简洁专业，像称职的秘书在早上汇报
- 今日日程是最重要的板块，放在最前面，确保会议信息完整
- 参会人超过5人时，显示前5人 + "+N人"（如"张三、李四、王五、赵六、钱七 +8人"）
- 今天没有会议时，"今日日程"显示"今天没有会议安排，可以专注处理待办事项。"
- "本周预览"在周五或无后续事件时省略
- 重点突出"需要回复或处理"的事项
- 某类数据为空则省略该板块（包括文件动态、本周预览）
- 不要重复原文，用自己的语言归纳
- "今日建议"要结合昨天的未完成事项 + 今天的日程给出具体可行的建议
