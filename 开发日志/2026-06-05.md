# 2026-06-05 Mac app 大图加载修复 - 跨平台 file_path 还原

## 背景

昨天发布的 Mac 端第一期(commit `ac36fa6`),实机测试发现:
- ✅ 网格缩略图能正常显示
- ✅ media 元信息(大小、尺寸、时长等)能显示
- ❌ 点开详情后大图/视频不显示

## 根因

`media.file_path` 字段在 backend 端存的是**写入时机器上的绝对路径**(例 `E:/MM/uploads/2026/06/04/xxx.mp4`),不是我最初 spec 里假定的相对路径。

证据链:
- `backend/app/config.py:40` — `UPLOAD_DIR = os.path.abspath(...)` → 绝对路径
- `backend/app/config.py:138` — `get_upload_dir() = os.path.join(UPLOAD_DIR, year, month, day)` → 仍是绝对路径
- `backend/app/services/media_service.py:56-60` — `dest_path = os.path.join(upload_dir, ...)` 写入 `Media.file_path`

Mac 上没有 E: 盘,直接当文件路径打开必然失败。
缩略图能出是因为它走 `data/thumbs/{id}.webp` 拼接,不依赖 `file_path`。

## 改动

### #1 `Media.localFileURL` 跨平台路径还原

`MyNote/MyNote/Models.swift`

新逻辑:
1. 把 `\` 归一化为 `/`(兼容 Windows 反斜杠)
2. 找 `/uploads/` 子串,截取从 `uploads/...` 起的相对部分
3. 拼到当前 `DATA_ROOT` 之下

覆盖场景:
- Windows 绝对:`E:/MM/uploads/2026/06/04/x.mp4` → `<DATA_ROOT>/uploads/...`
- POSIX 绝对:`/srv/mm/uploads/...` → 同上
- Windows 反斜杠:`E:\MM\uploads\...` → 同上
- 已是相对:`uploads/...` → 同上

理由:`uploads/` 是 backend 写入路径的稳定 marker(`get_upload_dir()` 强制结构),按它切分跨平台最稳。

### #2 详情视图加诊断信息

`MyNote/MyNote/MediaDetailView.swift`

图片加载失败时,占位区显示:
- 解析后的本地路径(让人看清 app 在找哪里)
- 原始 `file_path`(让人看清 backend 存的什么)
- `Text` 加 `.textSelection(.enabled)` 方便复制

避免下次再碰到类似问题时只看到"无法加载图片"这种无信息提示。

### #3 spec 同步更新

`openspec/changes/mac-native-local-db-media-grid/specs/mac-local-database/spec.md`

修正 "本地媒体文件路径解析" requirement:
- 说明 `file_path` 是绝对路径不是相对路径
- 加 Windows/POSIX/反斜杠三个场景

## 影响

- 只动 Mac 端 2 个 Swift 文件 + 1 个 spec,不影响 backend / Android / Vue / Electron
- 兼容向前:如果未来 backend 改成相对路径存,逻辑仍然成立(都会走 `uploads/...` 这条路径)

## 后续待办

依然在 `MyNote/MAC_TODO.md`,QuickLook、压测、回退验证等。

# 2026-06-05 Mac app 预览窗口左右切换

## 需求

详情窗口只能看单张,要回去网格再点下一张,效率太低。
要求:打开详情后能用键盘/按钮翻上一张/下一张,翻到列表末尾自动加载下一页。

## 改动

### #1 `MediaDetailView` 重构为列表感知

`MyNote/MyNote/MediaDetailView.swift`

接口从 `let media: Media` 改成:
- `mediaList: [Media]` — 当前列表快照
- `@Binding currentIndex: Int` — 当前索引,绑回外层
- `hasMore: Bool` + `onNeedMore: () async -> Void` — 末尾自动续接下一页

新增:
- 左右两侧浮动圆形按钮(`chevron.left/right`),禁用态半透明
- 隐藏的 keyboardShortcut 按钮承载 `←` / `→` / `ESC`
- header 显示 `当前/总数(+)`,`+` 表示后面还有未加载
- 视频切换时 `rebuildPlayer()` 释放旧 `AVPlayer`,自动播放新一个
- 末尾且 `hasMore` 时,`goNext()` 先 `await onNeedMore()` 再前进

### #2 抽出 `LocalLargeImageView`

同文件内 private struct。原实现在主线程同步 `NSImage(contentsOf:)`,大图会卡。
新实现:
- `Task.detached(priority: .userInitiated)` 后台解码
- `.task(id: media.id)` 切换时自动取消上一个、启动新的
- **不接入 `LocalImageLoader` 共享缓存** —— 大图 mediaId 会和缩略图 key 冲突互覆盖,500 张大图也吃内存

### #3 `MediaLibraryView` sheet 用法切换

`MyNote/MyNote/MediaLibraryView.swift`

- `selectedMedia: Media?` → `detailIndex: Int` + `showDetail: Bool`
- `.sheet(item:)` → `.sheet(isPresented:)`

理由:`.sheet(item:)` 在 item id 变化时会触发 sheet 重建动画;改成 isPresented 后,
左右翻页只改内部 index,sheet 容器不重建,翻页跟手且没有闪烁。

`ForEach` 改用 `Array(viewModel.media.enumerated())` + `id: \.element.id`,
拿到 index 写进 `detailIndex`。

## 设计取舍

- **没做无限循环翻页**:列表最后一张+无更多 → 右按钮禁用,直接结束体验。
  循环让人误以为有新内容,反而困惑。
- **末尾触发 loadMore 而不是预加载**:第一期保守,等用户实际行为再观察是否预拉。
- **没做手势滑动**:trackpad 双指滑要走 NSEvent,SwiftUI 上做得不优雅。
  键盘/按钮已经够用,未来真有需要再加 `.gesture(DragGesture)`。
- **视频切换默认 autoplay**:跟原来单视频 `VideoPlayer(player: AVPlayer(url:))` 行为一致,
  之前是隐式 autoplay,改成显式 `player.play()` 后行为不变。

Mac 上 Xcode 跑起来确认:
- [ ] 点缩略图打开详情,显示 `n / total(+)`
- [ ] `←` `→` 切换,图片/视频都能即时换

# 2026-06-05 Mac 预览体验优化:打开秒响应 + 窗口不跳

## 问题

刚做完左右切换后,实机用了一会发现两个糟糕的点:

1. **打开预览很慢**:从网格点一张图,先看到黑框 + ProgressView,过一阵子图才出来。
   感觉好像在加载好几秒。
2. **左右切换时窗口在跳**:不同比例的图片(4:3 vs 16:9 vs 竖屏)切换时,
   sheet 自己在变大变小,图片区域的位置也在跳。

## 根因

**慢:** 大图(几 MB JPEG)在主线程之外解码到 NSImage 仍然要几十到几百毫秒。
这段时间 `LocalLargeImageView` 显示的是空 ZStack + ProgressView,而不是已经在
内存里的缩略图。**网格里点开的那张,缩略图刚刚在 `LocalImageLoader.shared` 里命中过**,
按理应该秒级垫上去。

**跳:** sheet 用 `.frame(minWidth: 720, minHeight: 540)` 给最小尺寸,没给 ideal。
SwiftUI 的 sheet 会按内容 intrinsic size 决定初始大小,而 `Image.scaledToFit()`
没有外层确定 frame 约束时会按图片本身尺寸撑开 sheet 容器。结果就是:
4000×3000 的图把 sheet 撑大,切到 1920×1080 又缩回。

## 改动

### #1 `LocalImageLoader` 新增 `cached(mediaId:)` 只读接口

`MyNote/MyNote/LocalImageLoader.swift`

```swift
func cached(mediaId: Int) -> NSImage?
```

只查缓存不发起加载,给详情视图用。

### #2 `LocalLargeImageView` 双阶段加载

`MyNote/MyNote/MediaDetailView.swift`

`task(id:)` 流程改为两步:
1. **Step 1 同步占位**:先 `await LocalImageLoader.shared.cached(mediaId:)`,
   命中就立即显示缩略图(从网格点开必命中,actor await 实际是字典查找,~0 延迟)
2. **Step 2 后台解大图**:`Task.detached` 解原图,完成后替换占位

新增 `isPlaceholder` 状态:大图加载失败时,如果占位还在,就保留占位当兜底,
不回退到"无法加载"提示。

`Image` 加 `.interpolation(.medium)`,让低分辨率缩略图被拉大时不至于太糊。

### #3 窗口固定 ideal 尺寸

```swift
.frame(minWidth: 900, idealWidth: 1100, maxWidth: .infinity,
       minHeight: 600, idealHeight: 760, maxHeight: .infinity)
```

`ideal*` 让 sheet 初始就是 1100×760,不再被 Image intrinsic 撑开。
`max .infinity` 保留用户手动拉大窗口的能力。
`min` 防用户拉到看不清。

## 设计取舍

- **没做高斯模糊缩略图占位**:占位本身就是 320px 的低分辨率 webp,放大到 1100px
  会有点糊但能看清主体,够用了。再加 blur 反而不知道在加载什么。
- **`task(id:)` 不在 Step 1 之后 sleep/await Task.yield()**:有过这个想法 ——
  让 UI 先渲染一帧占位再开始解大图。但实测 `Task.detached` 已经不阻塞主线程,
  额外 yield 反而引入肉眼可感的延迟。
- **没把缩略图也缓存在 detail view 自己的 state**:NSCache 已经是全局的,
  detail 复用没有额外成本,key=mediaId 也不会和大图冲突(大图不写缓存)。
- **大图仍然不写 LocalImageLoader 缓存**:之前的考虑没变 —— 大图体积大,
  和缩略图共用 key 会互相覆盖。每次重开 detail 重新解大图,代价 ~100ms,
  接受。

## 待人工验证

- [ ] 网格点开任意图,瞬间看到缩略图(不再是黑框),~100ms 后替换为清晰大图
- [ ] 左右切换时,sheet 窗口大小**完全不变**
- [ ] 切到没在网格滚动过的图(直接打开列表第一张),仍然走 ProgressView 兜底
- [ ] 用户手动拉大窗口后,切换图片不会回弹尺寸

# 2026-06-05 多 Repository 媒体路径重构

## 背景

`media.file_path` 之前存的是**写入时机器上的绝对路径**(例 `E:\MM\uploads/2026/06/04/x.mp4`),带来三类问题:

1. DATA_ROOT 搬家就废 —— 路径硬编码到具体磁盘
2. 外接硬盘换盘符就废 —— Mac 上 WD-Blue 今天 `/Volumes/WD_BLUE`、明天 `/Volumes/WD_BLUE-1`
3. 跨平台读不通 —— Mac 上靠 `Models.swift` 字符串抠 `/uploads/` 子串兜底,STATIC_DIRS 里的原文件 Mac 完全打不开

Backend 端 `STATIC_DIRS` 这套"挂载目录"概念其实已经存在(`config.py` 的 `is_mounted_path` / `to_url_path`),只是 DB 里始终存绝对路径,挂载概念没形式化进 schema。这次把它正式落地。

## 设计要点

- DB 改存「相对挂载根」的 forward-slash 相对路径
- 新增 `media.repo_id`(TEXT, indexed, NOT NULL,默认 `uploads`)标识所在 repo
- 各端运行时按 `{repo_id: abs_mount_path}` 注册表拼绝对路径
- Backend 的 `UPLOAD_DIR` 和 `STATIC_DIRS` **一视同仁**,repo_id 都用 `basename(path).lower()`
- URL 前缀仍用 mount basename(`/Uploads/2026/...`),跟 FastAPI StaticFiles 一致
- Android **完全不参与** repo_id —— 它只认 backend 返回的完整 `file_url`,Gson 自动 ignore 新增的 `repo_id` 字段
- Mac 不引入 Security-Scoped Bookmark(app 未开 sandbox,普通 path string 够用)
- Mac onboarding 不变,自动种 `default repo = <DATA_ROOT>/uploads`

## 改动

### #1 Backend `config.py` 加 Repository 抽象

`backend/app/config.py`

新方法(老的 `to_url_path` / `is_mounted_path` 保留作 deprecated,供 alembic downgrade / scripts 用):

- `get_repositories() -> {repo_id: abs_path}` —— UPLOAD_DIR + STATIC_DIRS 各按 basename 聚合
- `default_repo_id() -> basename(UPLOAD_DIR).lower()`
- `resolve_to_absolute(repo_id, rel) -> abs path | None` —— 反向 join,forward-slash 转 os.sep
- `register_relative(abs_path) -> (repo_id, rel)` —— 按 mount 长度 DESC 匹配,处理嵌套 mount
- `url_for(repo_id, rel) -> "/{basename}/rel/path"`

### #2 `Media` model + Alembic 迁移

`backend/app/models/__init__.py:Media`:
```
repo_id = Column(String(64), nullable=False, default="uploads",
                 server_default="uploads", index=True)
```

新文件 `backend/alembic/versions/20260606_1200_e1f2a3b4c5d6_media_repo_id.py`

三步 upgrade:
1. 加 nullable `repo_id` 列
2. backfill —— **裸 SQL,不 import app.config**(否则 ffmpeg 探活会 sys.exit)。直接读 env 的 `UPLOAD_DIR` + `STATIC_DIRS`,按最长 mount 前缀匹配,重写 `file_path`。未命中 mount 的行标 `repo_id="__legacy__"`,`file_path` 保留绝对路径 + 日志告警
3. 收紧 NOT NULL + `server_default="uploads"` + 加 `ix_media_repo_id` 索引

可逆 downgrade:读 env 重建 mount 表,把 (repo_id, rel) 重新 join 成绝对路径,drop index + drop column。`__legacy__` 行跳过(已经是绝对路径)。

实测在本机 85 行 media 数据上跑:全部归 `uploads`,**0 个 `__legacy__`**,耗时 < 1s。

### #3 所有 service / router 改走 resolve_to_absolute

之前所有 `os.remove(media.file_path)` / `os.path.exists(media.file_path)` / `shutil.copy2(src, media.file_path)` 的地方,先 `config.resolve_to_absolute(media.repo_id, media.file_path)` 拿绝对路径再用。

涉及:
- `backend/app/services/media_service.py` —— `_merge_media_into` / `replace_media_file` / `rotate_media`
- `backend/app/routers/media.py` —— `delete_preview` / 删 media 源文件分支
- `backend/app/services/smart_tag_service.py` —— CLIP fallback 路径

`process_standalone_file()` 新增最后一步:`(repo_id, rel) = config.register_relative(file_path)` 写进 model。

`replace_media_file()` 扩展名变化时,新路径 `register_relative()` 再写回 `media.file_path`,加 `assert new_repo_id == media.repo_id`。

### #4 `MediaUrlMixin` 接 repo_id

# 2026-06-05 Repository 配置:env vars → repositories.json 单一来源

## 背景

昨晚刚做完的多 Repository 重构,DB 那侧已经干净了(`media.file_path` 改相对 + `repo_id`),但**注册表本身**两边各活各的:

- **Backend** 从 env vars `UPLOAD_DIR` + `STATIC_DIRS` 读(`start.bat:3-4`、`start.sh:5-6`)
- **Mac** 存 `UserDefaults.repositories`,需要在 Mac 上独立配一遍

Windows backend 和 Mac 共用同一份 SQLite,但两边物理盘符不同(Windows `F:\AV`、Mac `/Volumes/AV`),没有一份权威的"`repo_id` → 各平台路径"表。Mac 的 onboarding 也只是把 `<DATA_ROOT>/uploads` 自动种成 default repo,STATIC_DIRS 里的额外盘 Mac 永远不知道。

这次把注册表搬到 `<DATA_ROOT>/repositories.json`,一份文件 Backend + Mac 共享。**硬切换**,旧 env vars 直接删。

## 设计要点

- JSON 文件名:`<DATA_ROOT>/repositories.json`,跟 `db.sqlite3` 同级
- Schema 带 `version: 1` 留口子
- 每个 repo 一个 `paths` 子对象,key 是 `windows` / `darwin` / `linux`(Python `sys.platform` 与 Swift 概念都能直接对应)
- 路径用 forward-slash 写(Windows 也认),避免 JSON 转义反斜杠
- `default_repo_id` 显式声明新上传文件落在哪个 repo —— 替代旧 `UPLOAD_DIR` 的隐式语义
- `human_name` 可选,空 → UI 回退到 `repo_id`
- **URL prefix 顺手从 `/{basename(mount)}` 改成 `/{repo_id}`** —— `repo_id` 在 JSON 是规范的 key,直接拿来作 URL 段;basename 大小写在 Mac/Windows 上不一致,改完不再纠结
- **fail-fast 不容错**:文件缺失 / 解析失败 / 版本不匹配 / 当前平台缺路径 → `sys.exit(1)`
- **不在启动时检查 path 物理存在** —— 跟 Mac 端 `isAvailable` 语义对齐,允许外接盘没插也能启动,对应 media 走 404
- Mac `dataRoot` 继续留 UserDefaults(因为它是"去哪里读 repositories.json"的入口,不能进 JSON,否则循环)
- Android 不动 —— 走 backend HTTP,不需要本地路径

## Schema

```json
{
  "version": 1,
  "default_repo_id": "uploads",
  "repositories": {
    "uploads": {
      "human_name": "Uploads",
      "paths": {
        "windows": "E:/Note/Uploads",
        "darwin":  "/Volumes/Note/Uploads"
      }
```

### #5 Alembic 辅助 migration + 哨兵环境变量

新文件 `backend/alembic/versions/20260605_2100_f2a3b4c5d6e7_seed_repositories_json.py`

- `down_revision = 'e1f2a3b4c5d6'`
- `upgrade()`:检测 `<DATA_ROOT>/repositories.json` 已存在 → 跳过;不存在 → 按当前进程 env 的 `UPLOAD_DIR` + `STATIC_DIRS` 构造一份(只填当前平台 path,警告用户手动补另一平台)
- `downgrade()` no-op
- 刻意**不 import** `app.config`(同老 migration 模式),避免触发 ffmpeg 探活 + 新加的 fail-fast 读 JSON

**遇到的死结**:alembic `env.py` 加载时会 import `app.config` 拿 `get_db_path()`,而 config 模块现在 eager fail-fast 读 JSON —— 新机器跑 `alembic upgrade head` 时,seed migration 还没跑,模块加载就先 sys.exit 了。

解法是加哨兵 env var `ALEMBIC_SKIP_REPO_LOAD=1`:
- `alembic/env.py` 在 `from app.models` / `from app.config` **之前** `os.environ.setdefault("ALEMBIC_SKIP_REPO_LOAD", "1")`(必须在 import 前)
- `app/config.py` 模块尾部:`if os.getenv("ALEMBIC_SKIP_REPO_LOAD") != "1": _load_repositories(...)` —— 跳过 eager 加载,`_REPOSITORIES` 保持空 dict
- `app/config.py:validate_repositories()` 在哨兵模式下直接 `return`(否则 `app/__init__.py` 启动时会因空 dict 报错)

副作用:`app/utils/__init__.py:11` 的 import-time 副作用 `MEDIA_UPLOAD_DIR = config.get_upload_root()` + `os.makedirs(...)` 也会因空 dict 抛 KeyError。这个常量**全代码库没人用**,直接删掉(`process_file()` 内部本来就有自己的 `os.makedirs(upload_dir, exist_ok=True)`)。

## 行为变化(留意)

- **URL prefix 大小写改了**:`/Uploads/2026/...` → `/uploads/2026/...`(repo_id 小写)。客户端如果有缓存 `/Uploads/...` 的硬编码 URL 要清,但浏览器 fetch 在 Windows StaticFiles 下大小写不敏感 → 影响实际为 0
- **`__legacy__` 行不再有 fallback URL** —— 之前 `to_url_path` 把绝对路径当 URL 路径返出去(也是 404),现在直接返空串。前端见空 URL 应显示占位
- **首次升级机器**必须先跑 `alembic upgrade head`(自动 seed)或手写 `repositories.json`,否则 backend 起不来。Mac 端无文件 → `repositories` 为空,所有 media 走 `legacyExtractedURL` 兜底,降级到迁移前体验
