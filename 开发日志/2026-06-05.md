# 2026-06-05 Mac app 大图加载修复 - 跨平台 file_path 还原

## 背景

昨天发布的 Mac 端第一期(commit `ac36fa6`),实机测试发现:
- ✅ 网格缩略图能正常显示
- ✅ media 元信息(大小、尺寸、时长等)能显示
- ❌ 点开详情后大图/视频不显示

## 根因

`media.file_path` 字段在 backend 端存的是**写入时机器上的绝对路径**(例 `E:/MM/uploads/2026/06/04/xxx.mp4`),不是我最初 spec 里假定的相对路径。

证据链:
- `backend/app/config.py:40` — `UPLOAD_DIR = os.path.abspath(...)` → 绝对路径
- `backend/app/config.py:138` — `get_upload_dir() = os.path.join(UPLOAD_DIR, year, month, day)` → 仍是绝对路径
- `backend/app/services/media_service.py:56-60` — `dest_path = os.path.join(upload_dir, ...)` 写入 `Media.file_path`

Mac 上没有 E: 盘,直接当文件路径打开必然失败。
缩略图能出是因为它走 `data/thumbs/{id}.webp` 拼接,不依赖 `file_path`。

## 改动

### #1 `Media.localFileURL` 跨平台路径还原

`MyNote/MyNote/Models.swift`

新逻辑:
1. 把 `\` 归一化为 `/`(兼容 Windows 反斜杠)
2. 找 `/uploads/` 子串,截取从 `uploads/...` 起的相对部分
3. 拼到当前 `DATA_ROOT` 之下

覆盖场景:
- Windows 绝对:`E:/MM/uploads/2026/06/04/x.mp4` → `<DATA_ROOT>/uploads/...`
- POSIX 绝对:`/srv/mm/uploads/...` → 同上
- Windows 反斜杠:`E:\MM\uploads\...` → 同上
- 已是相对:`uploads/...` → 同上

理由:`uploads/` 是 backend 写入路径的稳定 marker(`get_upload_dir()` 强制结构),按它切分跨平台最稳。

### #2 详情视图加诊断信息

`MyNote/MyNote/MediaDetailView.swift`

图片加载失败时,占位区显示:
- 解析后的本地路径(让人看清 app 在找哪里)
- 原始 `file_path`(让人看清 backend 存的什么)
- `Text` 加 `.textSelection(.enabled)` 方便复制

避免下次再碰到类似问题时只看到"无法加载图片"这种无信息提示。

### #3 spec 同步更新

`openspec/changes/mac-native-local-db-media-grid/specs/mac-local-database/spec.md`

修正 "本地媒体文件路径解析" requirement:
- 说明 `file_path` 是绝对路径不是相对路径
- 加 Windows/POSIX/反斜杠三个场景

## 影响

- 只动 Mac 端 2 个 Swift 文件 + 1 个 spec,不影响 backend / Android / Vue / Electron
- 兼容向前:如果未来 backend 改成相对路径存,逻辑仍然成立(都会走 `uploads/...` 这条路径)

## 后续待办

依然在 `MyNote/MAC_TODO.md`,QuickLook、压测、回退验证等。

# 2026-06-05 Mac app 预览窗口左右切换

## 需求

详情窗口只能看单张,要回去网格再点下一张,效率太低。
要求:打开详情后能用键盘/按钮翻上一张/下一张,翻到列表末尾自动加载下一页。

## 改动

### #1 `MediaDetailView` 重构为列表感知

`MyNote/MyNote/MediaDetailView.swift`

接口从 `let media: Media` 改成:
- `mediaList: [Media]` — 当前列表快照
- `@Binding currentIndex: Int` — 当前索引,绑回外层
- `hasMore: Bool` + `onNeedMore: () async -> Void` — 末尾自动续接下一页

新增:
- 左右两侧浮动圆形按钮(`chevron.left/right`),禁用态半透明
- 隐藏的 keyboardShortcut 按钮承载 `←` / `→` / `ESC`
- header 显示 `当前/总数(+)`,`+` 表示后面还有未加载
- 视频切换时 `rebuildPlayer()` 释放旧 `AVPlayer`,自动播放新一个
- 末尾且 `hasMore` 时,`goNext()` 先 `await onNeedMore()` 再前进

### #2 抽出 `LocalLargeImageView`

同文件内 private struct。原实现在主线程同步 `NSImage(contentsOf:)`,大图会卡。
新实现:
- `Task.detached(priority: .userInitiated)` 后台解码
- `.task(id: media.id)` 切换时自动取消上一个、启动新的
- **不接入 `LocalImageLoader` 共享缓存** —— 大图 mediaId 会和缩略图 key 冲突互覆盖,500 张大图也吃内存

### #3 `MediaLibraryView` sheet 用法切换

`MyNote/MyNote/MediaLibraryView.swift`

- `selectedMedia: Media?` → `detailIndex: Int` + `showDetail: Bool`
- `.sheet(item:)` → `.sheet(isPresented:)`

理由:`.sheet(item:)` 在 item id 变化时会触发 sheet 重建动画;改成 isPresented 后,
左右翻页只改内部 index,sheet 容器不重建,翻页跟手且没有闪烁。

`ForEach` 改用 `Array(viewModel.media.enumerated())` + `id: \.element.id`,
拿到 index 写进 `detailIndex`。

## 设计取舍

- **没做无限循环翻页**:列表最后一张+无更多 → 右按钮禁用,直接结束体验。
  循环让人误以为有新内容,反而困惑。
- **末尾触发 loadMore 而不是预加载**:第一期保守,等用户实际行为再观察是否预拉。
- **没做手势滑动**:trackpad 双指滑要走 NSEvent,SwiftUI 上做得不优雅。
  键盘/按钮已经够用,未来真有需要再加 `.gesture(DragGesture)`。
- **视频切换默认 autoplay**:跟原来单视频 `VideoPlayer(player: AVPlayer(url:))` 行为一致,
  之前是隐式 autoplay,改成显式 `player.play()` 后行为不变。

## 待人工验证

Mac 上 Xcode 跑起来确认:
- [ ] 点缩略图打开详情,显示 `n / total(+)`
- [ ] `←` `→` 切换,图片/视频都能即时换
- [ ] 切换视频时上一个停掉,新的自动播
- [ ] 翻到列表末尾,右按钮触发加载下一页,加载完自动前进
- [ ] 第一张右键禁用,无更多内容时最后一张右键禁用
- [ ] `ESC` 关闭

# 2026-06-05 Mac 预览体验优化:打开秒响应 + 窗口不跳

## 问题

刚做完左右切换后,实机用了一会发现两个糟糕的点:

1. **打开预览很慢**:从网格点一张图,先看到黑框 + ProgressView,过一阵子图才出来。
   感觉好像在加载好几秒。
2. **左右切换时窗口在跳**:不同比例的图片(4:3 vs 16:9 vs 竖屏)切换时,
   sheet 自己在变大变小,图片区域的位置也在跳。

## 根因

**慢:** 大图(几 MB JPEG)在主线程之外解码到 NSImage 仍然要几十到几百毫秒。
这段时间 `LocalLargeImageView` 显示的是空 ZStack + ProgressView,而不是已经在
内存里的缩略图。**网格里点开的那张,缩略图刚刚在 `LocalImageLoader.shared` 里命中过**,
按理应该秒级垫上去。

**跳:** sheet 用 `.frame(minWidth: 720, minHeight: 540)` 给最小尺寸,没给 ideal。
SwiftUI 的 sheet 会按内容 intrinsic size 决定初始大小,而 `Image.scaledToFit()`
没有外层确定 frame 约束时会按图片本身尺寸撑开 sheet 容器。结果就是:
4000×3000 的图把 sheet 撑大,切到 1920×1080 又缩回。

## 改动

### #1 `LocalImageLoader` 新增 `cached(mediaId:)` 只读接口

`MyNote/MyNote/LocalImageLoader.swift`

```swift
func cached(mediaId: Int) -> NSImage?
```

只查缓存不发起加载,给详情视图用。

### #2 `LocalLargeImageView` 双阶段加载

`MyNote/MyNote/MediaDetailView.swift`

`task(id:)` 流程改为两步:
1. **Step 1 同步占位**:先 `await LocalImageLoader.shared.cached(mediaId:)`,
   命中就立即显示缩略图(从网格点开必命中,actor await 实际是字典查找,~0 延迟)
2. **Step 2 后台解大图**:`Task.detached` 解原图,完成后替换占位

新增 `isPlaceholder` 状态:大图加载失败时,如果占位还在,就保留占位当兜底,
不回退到"无法加载"提示。

`Image` 加 `.interpolation(.medium)`,让低分辨率缩略图被拉大时不至于太糊。

### #3 窗口固定 ideal 尺寸

```swift
.frame(minWidth: 900, idealWidth: 1100, maxWidth: .infinity,
       minHeight: 600, idealHeight: 760, maxHeight: .infinity)
```

`ideal*` 让 sheet 初始就是 1100×760,不再被 Image intrinsic 撑开。
`max .infinity` 保留用户手动拉大窗口的能力。
`min` 防用户拉到看不清。

## 设计取舍

- **没做高斯模糊缩略图占位**:占位本身就是 320px 的低分辨率 webp,放大到 1100px
  会有点糊但能看清主体,够用了。再加 blur 反而不知道在加载什么。
- **`task(id:)` 不在 Step 1 之后 sleep/await Task.yield()**:有过这个想法 ——
  让 UI 先渲染一帧占位再开始解大图。但实测 `Task.detached` 已经不阻塞主线程,
  额外 yield 反而引入肉眼可感的延迟。
- **没把缩略图也缓存在 detail view 自己的 state**:NSCache 已经是全局的,
  detail 复用没有额外成本,key=mediaId 也不会和大图冲突(大图不写缓存)。
- **大图仍然不写 LocalImageLoader 缓存**:之前的考虑没变 —— 大图体积大,
  和缩略图共用 key 会互相覆盖。每次重开 detail 重新解大图,代价 ~100ms,
  接受。

## 待人工验证

- [ ] 网格点开任意图,瞬间看到缩略图(不再是黑框),~100ms 后替换为清晰大图
- [ ] 左右切换时,sheet 窗口大小**完全不变**
- [ ] 切到没在网格滚动过的图(直接打开列表第一张),仍然走 ProgressView 兜底
- [ ] 用户手动拉大窗口后,切换图片不会回弹尺寸
