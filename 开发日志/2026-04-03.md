# 2026-04-03 开发日志

## 实现前后端双向同步架构

### 背景

之前的同步方案存在以下问题：
- 全量同步不可扩展（每次拉取全部数据）
- 后端没有 `POST /api/sync/apply` 端点（Android push 完全不工作）
- 缺少删除追踪（硬删除后客户端无感知）
- 无后台同步、无网络检测、无实时推送

### 实现内容

#### 后端

**新增 `SyncLog` 模型** (`backend/app/models/__init__.py`)
- 记录 `MESSAGE | ACTOR | MEDIA | TAG` 的 `UPSERT | DELETE` 操作
- 复合索引 `(timestamp, id)` + `(entity_type, entity_id)`
- 保留期 90 天

**新增 `sync_log_service.py`**
- SQLAlchemy `after_flush` 事件监听器：与业务数据在同一事务中原子写入
- 使用 thread-local 防止递归 flush
- 进程内 SSE 事件总线（`asyncio.Queue` 列表）
- `after_commit` 后广播通知给所有 SSE 订阅者

**新增 `routers/sync.py`** — 三个端点：
1. `GET /sync/changes?since={ISO}&limit=500`
   - 增量拉取，返回去重后的变更列表
   - UPSERT 附带完整实体快照（MESSAGE 含嵌套 media_items + tags）
   - since 为空/超过 90 天 → HTTP 410，客户端回退全量同步
2. `POST /api/sync/apply`
   - 按序处理 Android 推送的变更
   - Last-write-wins by `updated_at`，逐条 try/catch 不中断批次
   - camelCase → snake_case 字段映射
3. `GET /sync/events`
   - SSE 实时通知（仅携带元数据，不含完整数据）
   - 30 秒心跳保活

**修改 `app/__init__.py`**
- 启动时注册 SyncLog 监听器
- `Base.metadata.create_all(checkfirst=True)` 自动建表

#### Android

**新增依赖** (`build.gradle.kts`)
- `okhttp3:okhttp:4.12.0` + `okhttp3:okhttp-sse:4.12.0`（SSE）
- `androidx.work:work-runtime-ktx:2.9.1`（后台同步）

**扩展 `SyncService.kt`**
- 新增 `getChanges(since, limit)` 接口，返回 `Response<RemoteChangesResponse>`

**新增 `SyncModels`**
- `RemoteChangesResponse` / `RemoteChangeItem`
- `SyncResult.NeedFullSync`（410 响应时使用）

**`MessageRepository.syncIncremental(since)`**
- 增量拉取，处理 UPSERT（MESSAGE/ACTOR/MEDIA/TAG）和 DELETE
- has_more=true 时循环拉取
- 410 → 返回 `NeedFullSync`，调用方回退全量同步
- Gson 反序列化 `Map<String, Any?>` → 类型安全的 data class

**新建 `NetworkMonitor.kt`**
- `ConnectivityManager.NetworkCallback` 监听 WiFi 状态
- 暴露 `StateFlow<Boolean>` 供订阅

**新建 `SyncWorker.kt`**
- `CoroutineWorker`：push outbox → pull 增量（410 → 全量）
- `schedulePeriodicSync()`：每 15 分钟，WiFi constraint
- `scheduleImmediateSync()`：网络恢复时一次性触发
- `SharedPreferences` 存储 `last_sync_time`

**新建 `SseListener.kt`**
- OkHttp `EventSource` 连接 `/sync/events`
- 500ms debounce 触发增量同步
- 指数退避重连（2s → 30s）

**`MainActivity.kt`**
- 启动时调度定期同步
- 监听网络恢复触发即时同步

**`DatabaseManager.kt`**
- 注入 `NetworkMonitor`

#### Vue

**新建 `useSyncEvents.ts`**
- `EventSource` 连接 `/sync/events`
- 自动重连（指数退避）
- 回调接收 `(entityType, entityId, operation)`

## 后端 Bug 修复与性能改进

### 背景

对照 ARCHITECTURE.md 深度分析后端代码，发现多处 bug 和性能问题，统一修复。

### 修复内容

#### B-1 | sync/apply 批量原子性（严重）

- **文件**：`backend/app/routers/sync.py`
- **修复**：将 `POST /api/sync/apply` 的逐条 `db.commit()` 改为整批单一事务，全部成功才 commit，任意失败则全量 rollback。

#### B-3 | 大文件哈希碰撞（严重）

- **文件**：`backend/app/utils/__init__.py`
- **修复**：>100MB 文件改为采样首尾各 4MB 计算 blake2b（并前缀加文件大小），不再直接用文件大小作为 hash，消除大文件哈希碰撞导致误判重复的问题。

#### B-4 | 游标分页同 timestamp 跳行（高）

- **文件**：`backend/app/routers/sync.py`、`backend/app/schemas/sync.py`
- **修复**：`GET /sync/changes` 改为 `(timestamp, id)` 复合游标：
  - 查询条件：`(timestamp > t) OR (timestamp = t AND id > last_id)`
  - 新增 `since_id` 请求参数和 `next_cursor_id` 响应字段

#### B-6 | N+1 查询（高）

- **文件**：`backend/app/routers/actor.py`
- **修复**：演员列表页改用 `GROUP BY` 子查询批量获取 `message_count`；演员详情页同理批量获取各消息的 `media_count`，由 O(N) 降为 O(1)。

#### B-7 | 缺少复合索引（中）

- **文件**：`backend/app/models/__init__.py`
- **新增**：`Index("ix_mm_created_at_position", "created_at", "position")` — 支持媒体网格复合游标分页的高效过滤。

#### B-8 | SSE 订阅队列内存泄漏（中）

- **文件**：`backend/app/services/sync_log_service.py`
- **修复**：
  - `_subscribers` 列表操作加 `threading.Lock` 保护
  - `asyncio.Queue` 加 `maxsize=100`
  - `_broadcast` 中 `put_nowait` 失败（队列满）时自动将孤立 queue 从订阅列表清除

#### B-9 | 数据完整性约束缺失（中）

- **文件**：`backend/app/models/__init__.py`
- **新增约束**：
  - `Tag.name` — `UniqueConstraint("uq_tag_name")`
  - `Actor.name` — `UniqueConstraint("uq_actor_name")`
  - `MessageMedia(message_id, position)` — `UniqueConstraint("uq_mm_message_position")`

#### B-13 | ffmpeg 路径默认值无警告（低）

- **文件**：`backend/app/config.py`、`backend/app/__init__.py`
- **修复**：新增 `AppConfig.check_paths()` 方法，启动时检查 `FFMPEG_PATH`/`FFPROBE_PATH` 是否存在，不存在则打 warning 并告知用户通过环境变量设置。
