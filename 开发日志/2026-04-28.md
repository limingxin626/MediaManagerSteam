## 消息中超链接强制走默认浏览器

### 背景

Electron 包装下，消息正文渲染的 markdown 链接点击后默认会在 BrowserWindow 内导航，覆盖整个应用。需要让所有 http(s) / mailto 链接统一在系统默认浏览器中打开。

### 实现内容

**Electron — `electron/main.js`**

- 顶部 import 加 `shell`
- `createWindow()` 内为 `win.webContents` 注册：
  - `setWindowOpenHandler`：`target="_blank"` 触发的弹窗，命中 `http(s)://` / `mailto:` 时调 `shell.openExternal` + 返回 `{ action: 'deny' }`
  - `will-navigate`：拦截同窗口跳转，目标是外部协议且不等于当前 URL 时 `event.preventDefault()` + `shell.openExternal`
- `isExternal` 用 `/^(https?:|mailto:)/i` 判断，避免误伤 `file://` / 应用内 hash 路由

**前端 — `vue/src/utils/markdown.ts`**

- 包装 `marked.Renderer().link`，给生成的 `<a>` 加 `target="_blank" rel="noopener noreferrer"`
- 浏览器 / PWA 场景下也是浏览器新标签打开；Electron 下走上面的 `setWindowOpenHandler` 转交系统默认浏览器
- `renderMarkdown` 调用改成 `marked.parse(normalized, { renderer })`

### 涉及文件

- `electron/main.js`
- `vue/src/utils/markdown.ts`

### 验证

- `pnpm exec vue-tsc --noEmit` 通过
- 端到端未手测，下次启动后请按以下手测：
  1. 普通浏览器访问消息：点链接，新标签打开，原页签留在应用
  2. Electron 启动应用进入消息：点链接，应用窗口不变，系统默认浏览器自动弹出新标签打开链接
  3. 点应用内站内链接（hash 路由 / `/media/123` 等）正常路由跳转，不被外开

## Message Tag 显式绑定，移除文本自动解析

### 背景

之前 message 的 tag 来源有两个：后端从 `text` 用正则 `#[\w\u4e00-\u9fff/\-]+` 自动解析、以及前端 TagPickerPopover 手动选择。文本解析会把 URL 里的 `http://x/y#section`、Markdown anchor、英文 `#` 等误判成 tag，且后端 `merge=False` 全量替换 message.tags 还会覆盖手动加的 tag。

新行为：tag 与文本解耦——只有用户在输入时通过 `#` 自动补全**显式选中**的 tag 才算绑定，其余 `#xxx` 文本仅作普通文本保留。已绑定 tag 在输入对话框上方以 chip 列表独立展示。Android sync (`/sync/messages`) 和 flomo 导入仍走文本解析（`MessageCreateFromClient` / sync.py / import_flomo.py 不动），保持兼容。

### 实现内容

**后端 — `backend/app/schemas/message.py`**
- `MessageCreate.tag_ids` 由 `List[int] = []` 改为 `Optional[List[int]] = None`，从而能区分"未传"和"明确空"
- `MessageUpdate` 新增 `tag_ids: Optional[List[int]] = None`

**后端 — `backend/app/routers/message.py`**
- `create_message`：`tag_ids is not None` 时跳过 `sync_tags_from_text`，直接 `message.tags = 查询结果`；`is None` 时走旧文本解析（兼容老客户端）
- `update_message`：同样逻辑——`tag_ids is not None` 时显式替换；`text` 单独 patch 但未带 `tag_ids` 时仍兜底文本解析
- `MessageCreateFromClient` / sync.py / import_flomo.py 都不动

**前端 — `vue/src/composables/useTagAutocomplete.ts`**
- 接口新增可选回调 `onTagPicked?: (tag: TagItem) => void`
- `selectTag` 不再向 textarea 插入 `#tagName `；改为：删除从 `currentTagStart` 到光标的 `#query` 触发片段，调用 `onTagPicked(tag)`，光标停在 `#` 原位置

**前端 — `vue/src/components/MessageComposeDialog.vue`**
- 新增 `selectedTags: TagItem[]` 状态
- 新增 prop `initialTags?: TagItem[]`，编辑模式打开时初始化为消息已绑 tag；创建模式如带 `tagId` prop 则预选
- textarea 上方新增 chip 行，每个 chip 显示 `#name` + ×按钮调 `removeTag`
- `useTagAutocomplete` 传入 `addTag` 作为 `onTagPicked` 回调；选中后立即出现在 chip 行
- `emit('updated', ..., tagIds)` 新增 tagIds 参数；create 路径的 `tag_ids` 改为来自 `selectedTags`

**前端 — `vue/src/views/Message.vue`**
- 新增 `dialogInitialTags` ref，`openEditDialog` 时从 `msg.tags` 拷贝
- 模板把 `:initial-tags="dialogInitialTags"` 传入 dialog
- `onDialogUpdated` 多收一个 `tagIds`，PATCH 请求体加上 `tag_ids: tagIds`

## MessageCard 移除文本内 `#hashtag` 高亮

### 背景

上一项把 tag 与文本解耦后，message 列表卡片仍在文本里把 `#xxx` 包成 `<span class="hashtag-inline">` 紫色高亮，并且 `visibleTagChips` 还会过滤掉文本中已出现的 tag chip，避免重复显示。新模型下文本里的 `#` 只是普通字符，应当去掉这套特殊处理：所有绑定 tag 一律走下方 chip 列表，文本原样渲染。

### 实现内容

**前端 — `vue/src/components/MessageCard.vue`**
- `renderedText`：去掉对 `#xxx` 的正则替换，直接 `renderMarkdown(text)`
- `visibleTagChips` 计算属性删除，模板改用 `messageTags` 直接渲染所有 tag chip
- 同步清理：`textRef`/`isTextClamped`/`checkTextClamped`/`watch`/`nextTick` 不再需要，连同 `<div ref="textRef">` 的 ref 一起删

**前端 — `vue/src/style.css`**
- 删除 `.hashtag-inline` / `.dark .hashtag-inline` 样式（已无引用）
