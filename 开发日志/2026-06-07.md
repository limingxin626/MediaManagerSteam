# 2026-06-07 Mac 媒体页修两个 bug:切 tab 丢状态 / 网格样式瑕疵

## 背景

页面骨架立起来后,发现媒体页两个体验问题:

1. 在媒体网格里滚到中间 → 切到「主页」或「消息」→ 切回 → 网格回到顶部,选中项也丢了,刚加载好的远期桶又重新加载一遍
2. 媒体网格区域有蓝色 focus ring 描边(看起来双层),右侧还有一根垂直滚动条 —— 都不像 Finder / Photos

详见 `openspec/changes/mac-media-page-state-and-styling/`。

## 改动

### 1. `ContentView`:`switch` 改 `ZStack`

把 `private var content: some View { switch selectedTab { ... } }` 整个换成 `ZStack { HomePlaceholderView(); MessagesPlaceholderView(); MediaLibraryView() }`,每个 view 用 `.opacity(selectedTab == .case ? 1 : 0)` + `.allowsHitTesting(selectedTab == .case)` 控可见性与交互。

三个 view 常驻 view 树后,SwiftUI 不会销毁/重建,`MediaLibraryView` 的 `@StateObject` 不会被重新初始化,`NSScrollViewBridge` 创建的 `NSScrollView` 也保留,`scrollTop` / `bucketCache` / 选中态都跟着活下来。

### 2. `MediaLibraryView`:`.focusEffectDisabled(true)`

`.focusable(true).focused($gridFocused).onKeyPress { ... }` 链上追加 `.focusEffectDisabled(true)`(macOS 14+)。键盘焦点保留、方向键/空格/ESC 仍然工作,只是不再画系统默认的蓝色 NSFocusRing;选中态用 `Color.accentColor` 描边自己起指示作用。

### 3. `NSScrollViewBridge`:`hasVerticalScroller = false`

`makeNSView` 里 `sv.hasVerticalScroller = true` 改为 `false`,`autohidesScrollers` 保持 `true`(对未来再开滚动条时冗余有效),`scrollerStyle = .overlay` 保持不变。

滚轮/trackpad/键盘 PgUp/PgDn 等手势注册在 `clip view` 上,与 scroller 开关无关,滚动行为完全保留。

## 关键设计决策

1. **ZStack 保 view 树,而不是把 VM 提到 App 级别单例** —— 主页/消息是 PlaceholderView,常驻零成本;媒体页 view 树存活,VM 自然跟着存活。无需在 App 入口提前实例化 VM,避免 VM 初始化与数据目录切换(`changeDataRoot` 走 `NSApplication.shared.terminate(nil)`)流程打架
2. **`.focusEffectDisabled` 而不是换 focus 机制** —— 关闭可见描边即可,键盘事件链完全保留,改动最小
3. **不换成 SwiftUI ScrollView** —— VirtualScrollView 的程序化跳转(`clip.scroll(to:)`)依赖 NSScrollView 的精确控制,改成 SwiftUI 滚动会失去跳转能力
4. **占位页与媒体页在 ZStack 中各自独立 `allowsHitTesting`** —— 避免隐藏的 view 拦截当前 tab 的点击事件

## Xcode 集成注意

老问题:`ContentView.swift` 改动后,SourceKit 经常报几个「Cannot find X in scope」的索引延迟(实际类型都在同 target),build 一次或 clean 重建即恢复。本次同样命中,在 MediaLibraryView / ContentView 上都看到一过性报错,以 `xcodebuild` 真实输出为准。

# 2026-06-08 Mac 消息流 prefetch 改造:vue 同款距离 N 条 + 滚动保留

## 背景

首版消息流的「加载更早」触发用的是「最顶部哨兵 `.onAppear」`,实际跑下来有两个问题:

1. **必须滚到最旧才触发**:用户得先滚到列表最顶部才开始拉下一页,体验上像「到底了才加载」,没有「快到底时就开始预加载」的聊天 App 预期
2. **prepend 后视口跳变**:vue 端有 `tryRestoreScroll` 用 scrollDelta 稳定视口(`vue/src/views/Message.vue:968-969`),Mac 端 prepend 20 条旧消息后用户当前看到的内容会突然跳 20 条位置

Vue 端的做法是 `PREFETCH_DISTANCE = 10`:在「距离顶/底 10 条」的元素上挂 IntersectionObserver,元素进入视口就触发 fetch。本期把 vue 的「距离 N 条」+「prepend 滚动保留」+「日期跳转抑制」三件套一起搬到 Mac。

## 改动

### 触发机制(从 sentinel 改 marker)

`MyNote/MyNote/MessagesView.swift`:

- **`loadMoreSentinel`**:删 `.onAppear`(原来 214-219 行),保留视觉(「加载更早…」spinner / 「已经是最早的消息了」)。它现在纯视觉占位条
- **`messagesScrollView`**:改用 `ForEach(viewModel.messages.indices, id: \.self)`,在 `idx == prefetchIndex` 位置(`min(10, count - 1)`)插入不可见 `Color.clear.frame(height: 1).onAppear { handlePrefetchTrigger() }` 触发器
  - **关键**:不给 marker 加 `.id(...)`。让 SwiftUI 用默认 identity,prepend 后旧 marker view 被销毁、新 marker view 实例化 → `onAppear` 重触发
- **`prefetchIndex`** computed property:对齐 vue `Math.min(10, els.length - 1)`,count < 10 时 marker 落在最后一条之前

### 滚动位置保留(vue `tryRestoreScroll` 等价)

- 每个 `MessageCard` 的 `.background(GeometryReader { ... })` 通过 `MessageFramePreferenceKey` 报告自己的 `(id, frame.minY)` 到父级
- 父级 `MessagesView` 用 `onScrollGeometryChange(for: ScrollGeometrySnapshot.self) { ... }` 拿 `contentOffset.y` + `containerSize.height`,在 action 里调 `computeTopmostMessageId()` 找「`frame.minY >= scrollTop + 1` 的最小值」对应的 id —— 这就是当前 viewport 顶部那条 message
- prefetch marker `onAppear` 触发 `loadMore` 前,把 `topAnchorMessageId` 通过 `restoreAnchor:` 参数传给 VM
- VM `loadMore` 把 anchor 写到 `prependedPendingAnchor: Int?`(@Published)
- View 的 `.onChange(of: viewModel.messages.count)`:先检查 `consumeScrollToBottomPending`(首屏到底),再检查 `consumePrependedPendingAnchor()`(prepend 恢复)—— 两条路径互斥,不会冲突
- 用 `proxy.scrollTo(anchor, anchor: .top)` 把视口稳定在用户原本在看的 message

### VM 加固(`MessagesViewModel.swift`)

新增字段:
- `@Published private(set) var prependedPendingAnchor: Int?` —— prepend 完成后 View 要 scrollTo 的 id
- `@Published private(set) var dateJumpPending: Bool` —— scrollToDate 后 ~500ms 内为 true,期间 prefetch 拒绝触发
- `private var requestGeneration: Int` —— loadInitial / selectFilter / scrollToDate 入口 +1;loadMore 入口捕获 `myGen`,await 结束后若不一致 → 丢弃结果(防止切过滤时旧 cursor 的 stale 数据污染新 filter)
- `private var lastLoadMoreAt: Date?` —— 200ms debounce(LazyVStack 复用 marker view 多次 onAppear 兜底)
- `private var dateJumpClearTask: Task<Void, Never>?` —— 500ms 后清 `dateJumpPending`

改动:
- `loadInitial` / `selectFilter` / `scrollToDate` 入口 `requestGeneration += 1`
- `scrollToDate` 设 `dateJumpPending = true` + 起 500ms 清 task
- `loadMore(restoreAnchor: Int? = nil)` 加 200ms debounce + dateJumpPending guard + generation check + 把 anchor 写到 `prependedPendingAnchor`
- 新增 `consumePrependedPendingAnchor() -> Int?`

### 边界 case 处理

| Case | 行为 |
|---|---|
| 首屏加载(20 条,count > 10) | scrollToBottomPending 触发,用户落底;marker 在 [10] 位置,不可见,onAppear 不触发 |
| 上滚到 [10] 附近 | marker onAppear → loadMore(记录 anchor)→ 成功后 scrollTo(anchor) 恢复视口 |
| prepend 后 marker 还在 [10] | SwiftUI 重建 marker view,onAppear 重触发下一批;200ms debounce 兜底 |
| 极快连续上滚 | `isLoading` 守门 + 200ms debounce 兜底 |
| `hasMore = false` | VM guard 直接返回 |
| `scrollToDate` 后立刻上滚 | 500ms 内 `dateJumpPending = true`,prefetch 拒绝触发 |
| 切过滤/日期跳转 在飞 loadMore 中发生 | `myGen != requestGeneration` → 丢弃结果 |
| 切 tab 来回 | `@StateObject` 持有 VM,所有内部状态保留;首次 onAppear 不重新触发 |
| count < 10 | `prefetchIndex = count - 1`,marker 落在最后一条之前;首屏后可能立即触发 → 正常,等 fetch 决定有没有更早 |

## 不变量

- `prefetchIndex` 与 vue `PREFETCH_DISTANCE = 10` 严格对齐
- 滚动保留算法与 vue `tryRestoreScroll` 同语义(用 id 而不是绝对 Y,因为 `ScrollViewProxy.scrollTo` 不支持坐标)
- `dateJumpPending` 500ms 窗口是经验值,够长能覆盖「日期跳转 → 用户立刻上滚」的 99% 场景
- `requestGeneration` 保证 in-flight `loadMore` 不会污染 `loadInitial` / `selectFilter` 后的 messages
