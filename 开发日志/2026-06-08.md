# 2026-06-08 Mac 消息页 inline markdown 渲染

## 背景

Mac 端消息页(`MessageCard` / `MessageDetailPane`)的正文一直走的是 `Text(AttributedString)` + `NSDataDetector` 仅做裸 URL 检测,markdown 标记(`**粗体**` / `*斜体*` / `` `行内代码` `` / `[label](url)`)直接以字面字符出现,与 vue 端用 `marked` 全量渲染的体验差很多。

本次需求:补齐 markdown 渲染,但**仅 inline 范围**(粗体 / 斜体 / 行内代码 / 链接 / 删除线),块级语法(标题 / 列表 / 引用 / 代码块)暂不展开,按字面保留。feed 卡片不做截断 / 折叠。

## 改动

### 新文件

- `MyNote/MyNote/MessageTextRenderer.swift` — `enum MessageTextRenderer` 静态工具:
  - `render(_ text:) -> AttributedString` — 解析 inline markdown + 兜底裸 URL 检测
  - 用 SwiftUI 原生 `AttributedString(markdown:options:)`,选项 `interpretedSyntax = .inlineOnlyPreservingWhitespace`:
    - inlineOnly:块级语法字面保留,不展开
    - preservingWhitespace:换行不被压缩(消息正文里的换行视觉上保留)
  - `failurePolicy = .returnPartiallyParsedIfPossible`:解析失败时尽量返回部分结果,不掉内容
  - 解析后再跑一遍 `NSDataDetector` 给"还没被 markdown 解析成 link 的裸 URL"上色挂 link;已经是 markdown link 的 run 跳过(不覆盖 `[label](url)` 的目标 URL)

### 改造

- `MyNote/MyNote/MessageCard.swift` — `textBody(_:)` 改用 `MessageTextRenderer.render(text)`,删除私有 `detectLinks` 实现(-18 行)
- `MyNote/MyNote/MessageDetailPane.swift` — 同款改动(-14 行)

### 不动

- Xcode project: 项目用的是 `PBXFileSystemSynchronizedRootGroup`(Xcode 16 file system synchronized folders),`MyNote/MyNote/` 下新文件**自动加入 target**,不需要手动改 xcodeproj
- vue 端、Android 端、backend 端零改动

## 设计决策

### 为什么不引入第三方完整 markdown 库

vue 端用的是 `marked`,渲染出 H1-H6 / 列表 / 引用 / 代码块 / 表格的 HTML 然后 v-html。SwiftUI 想达到同款效果需要:
1. 引第三方库(swift-markdown 解析,自己写 SwiftUI renderer)
2. 或者用 WebView 内嵌 HTML

两种都比"补齐 inline"重得多。用户选择仅 inline 范围,理由是消息正文以日常笔记为主,inline 标记(粗体强调 / 行内代码 / 链接)是高频用法,块级语法用得少,且 SwiftUI 原生 `AttributedString(markdown:)` 零依赖、可控。

### 为什么继续保留裸 URL 检测

`AttributedString(markdown:)` 只把符合 `[label](url)` 或 `<url>` 规范的当成 link,直接贴的 `https://example.com` 不会变蓝。NSDataDetector 兜底能保住裸 URL 可点击的体验(与之前一致)。

### 为什么不在 feed 卡片里折叠

vue 端做了 `line-clamp-10`,但 SwiftUI 的 `.lineLimit(10)` 对 inline markdown 的 AttributedString 高度估算不稳定,而且 Mac 屏幕宽,即便长内容自然滚也不影响 feed 浏览体验。保持现状不折叠是 SwiftUI 模式下的标准做法。
