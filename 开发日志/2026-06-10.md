# 2026-06-10 Mac 消息 markdown 渲染:段内单换行修复

## 问题

用户反馈两件事:

1. **换行太严格**:很多视觉上分两行(行间无空行)的内容,渲染后被拼到一起 —— 看起来像一行
2. **跨段选择不通**:鼠标拖选时,只能选当前一段,相邻段落选不进去

## 根因

`MarkdownBody.swift` 走 swift-markdown-ui,严格 CommonMark 语义把段内单换行(soft break)当作空格,
所以"两行连在一起"。vue 端 `marked` 设置了 `breaks: true`,把单换行当成 `<br>`,二者行为不一致。

跨段选择不通是 swift-markdown-ui 的根本限制:每个 block(段落/标题/列表/引用/代码块)
独立渲染成 SwiftUI view,`.textSelection(.enabled)` 不能跨 view 选。

## 本期改动:仅修问题 1

`MarkdownBody.swift` 新增 `softBreakToHardBreak(_:)` 静态方法:
- CRLF / CR 归一化为 `\n`
- 围栏代码块(``` / ~~~)区间内原样保留,避免破坏代码
- 段内单换行追加两空格(CommonMark 硬换行语法),保证渲染为 `<br>`
- 已有硬换行标记(行尾两空格 / 反斜杠)或后续是空行 / 当前是空行的不动

`body` 内 `Markdown(text)` 改成 `Markdown(Self.softBreakToHardBreak(text))`,其它样式不动。

### 副作用:问题 2 部分缓解

修了问题 1 之后,原本"两行被错误连成一段"的内容现在变成同一段内的两行 —— 这种情况选择能跨行。
但真正不同段落(中间有空行)之间还是选不到一起。

## 没做的事:NSTextView 桥接重写

跟用户确认后**暂缓**。彻底解决问题 2 需要把 `MarkdownBody` 整个换成 `NSViewRepresentable` 包的
`NSTextView`(只读 + 可选择 + 自动高度),所有 block 走单一 `NSAttributedString`。

确认当前 markdown 渲染**没有任何业务交互**:
- 链接:仅系统默认 OpenURLAction(点了用默认浏览器打开),没有自定义拦截
- 任务列表 checkbox:仅展示,不可勾选
- 图片:不可点开
- 代码块:无复制按钮
- 表格:不可排序选行

所以 NSTextView 重写没有要保留的交互,代价主要在样式:
- 围栏代码块圆角底色 → 降级成方角(`NSAttributedString.backgroundColor` 是按字符画的)
- 引用块竖线 → 改成左缩进 + 灰字,或用前缀字符 hack
- 表格 / 任务列表 → 退化成纯文本

下次需要时再做。

# 2026-06-10 (下午) Mac 消息预览翻页第二次失效

## 现象

消息卡里点开某张媒体进入预览后,左右键 / 翻页按钮**只能在最初点开的那张
和它的相邻一张之间来回切**,没法继续往后走。例如点第 1 张 → 右键到第 2 张
→ 再按右键无反应,只能按左键回到第 1 张。

## 根因

`MessagesView.previewState: PreviewState?` 是 `@State`,持有一个引用类型
`IndexBox` 包装的当前索引。传给 `MediaDetailView` 的 `Binding<Int>` 是
`Binding(get: { box.value }, set: { box.value = $0 })`。

问题:`Binding.set` 改 `box.value` 是**纯引用写入**,SwiftUI 不会观察到
`previewState`(值类型 struct)发生变化,**不会触发 `MessagesView` body
重算**。`MediaDetailView` 自身的 `@Binding currentIndex` 第一次写时它的
view 自身 invalidate + 重算把新值显示出来;但接下来 `onChange(of: currentIndex)`
比较新旧值时,`_currentIndex` 内部缓存的旧值实际上是上次 set 之前的状态,
加上父没重算导致的闭包捕获时序问题,第二次 set 之后 SwiftUI 内部依赖图
错位,翻页不再生效。

媒体库页(`MediaLibraryView`)同样写法没复现,**因为它用 NavigationStack push
推 `MediaPreviewDestination`,detail view 是独立的 navigation 子页**,
不依赖父 view 重算来传递新 binding 值。消息预览是 overlay 模式,所在 view 树
和父共享同一次 body 计算,所以暴露出来。

## 修法

- 删掉 `PreviewState.indexBox`
- 新增 `@State private var previewIndex: Int = 0` 持当前索引
- `MediaDetailView` 直接收 `$previewIndex`
- `openPreview` 时先 `previewIndex = safe` 再赋 `previewState`
- `closePreview` 不再从 `state.indexBox` 抄值(本来这条路径也没被消费)

SwiftUI 依赖追踪恢复正常,翻页连续可用。

`.id(state.id)` 仍然挂在 `MediaDetailView` 上,只在 session 切换(打开新预览
session、关闭再开)时变,翻页不会触发它重建 → 内部 AVPlayer / FocusState /
chromeVisible / mouseMonitor 不丢。

## 不动的

`MediaLibraryView` 的 `IndexBox` 写法保持原样 —— NavigationStack push 模式下
确实工作,改它无收益、有回归风险。`IndexBox` / `MediaPreviewDestination`
依然被那条路径使用,不能删。

## 改动文件

- `MyNote/MyNote/MessagesView.swift`:+10 / -15

commit `b21354d`。

# 2026-06-10 (晚) 媒体预览键盘翻页改走 AppKit local monitor

## 现象

按钮翻页(左右两侧 chevron)是好的;**键盘左右键不稳**:
- 有时按了不动 → 「翻不动」
- 有时一按跳两张 → 「会跳」
- 与媒体类型无关,纯图片也复现(所以不是 AVKit transport bar 抢左右键的问题)

## 根因

`MediaDetailView` 用 SwiftUI `.onKeyPress` + `@FocusState isFocused` 监听键盘。
overlay 模式下:
- 切换媒体时 `onChange(of: currentIndex)` 触发 content 子树重建,
  `LocalLargeImageView` / `VideoPlayer` 重新挂载过程中焦点链状态不稳,
  `@FocusState` 偶发被吞 → 下次 keypress 找不到接收者 → 「翻不动」
- 部分场景同一事件既被 `.onKeyPress` 接(SwiftUI 路径)、又被其它 NSResponder
  处理(AppKit 路径) → 「跳两项」

## 修法

加 `keyMonitor` (AppKit `NSEvent.addLocalMonitorForEvents(matching: .keyDown)`),
在 SwiftUI 焦点分发之前抢拦截:

- keyCode 123/124 (左右) → `goPrev` / `goNext`
- keyCode 53/49 (Esc/Space) → `closePreview`
- 一律 `return nil` 吞事件,杜绝下游再消费
- `isARepeat` 不响应,长按不连翻(Photos.app / 浏览器图片预览同款行为)
- `onAppear` 装、`onDisappear` 卸,跟 `mouseMonitor` 同一套模式

`.onKeyPress` 整段删除,避免与 monitor 双触发。`.focusable(true)` / `.focused`
保留 —— 给鼠标选择文本等语义用,跟键盘处理已无关。

## 局限

local monitor 是 app 进程级的,装上期间会无差别拦截所有 keyDown 事件
(包括跑到媒体库页的同款 detail view)。但因为只在 `onAppear → onDisappear`
窗口内有效,实际只在预览打开期间生效,关掉就还回去 —— 不会污染其它页面。
如果将来出现两个 `MediaDetailView` 并存的场景,需要加焦点判断;现在不存在,
不预防。

commit `fc596db`。

# 2026-06-10 (深夜) Mac 消息卡:行内展开式编辑(只读 ↔ 编辑两态)

## 背景

Mac 端消息卡到目前为止是 read-only —— 星标只能看不能点,文字 / tag 完全不可改。用户提了
两个候选方案:① 卡片直接变编辑框(永久编辑态);② 全屏页面编辑。讨论后选了**第三个**:
**行内展开**(GitHub issue 评论那种 —— 默认只读,点「编辑」原地展开成编辑器,Save / Cancel
固定底栏,其他卡片不受影响)。

理由:
- ① 永久编辑态:feed 滚动易误触,tag / 媒体编辑控件常驻挤压密度
- ② 全屏页面:对「改个错字」过重,丢 feed 上下文
- ③ 行内展开:浏览体验不变,编辑是显式动作;改动量适中

## 写入路径

走 backend `PATCH /messages/{id}`,**不**改 GRDB 为读写。理由:
- 复用 `message_service` 的 #hashtag 解析、SyncLog 落表逻辑,避免在 Mac 端重写一遍
- 与 Android sync / web 端 PATCH 走同一条路径,行为一致

对应改:`APIClient.updateMessage` 由 PUT 修正为 PATCH(后端实际接的是 PATCH),并加
`starred` 字段 —— 后端 `MessageUpdate` schema 早就支持 `starred / text / tag_ids` 同接口,
不需要单独 endpoint。

## VM 接口

`MessagesViewModel` 由 read-only 改成开放三个字段写:

- `commitEdit(messageId, text, tagIds) -> Bool`:返回是否成功,失败设 errorMessage 并保留
  编辑态(用户可重试 / 取消)。tagIds 传 nil = 不动 tag(后端按 text 重抽 #hashtag);
  传 [] = 清空;传非空 = 替换。
- `toggleStar(messageId)`:**乐观更新**——先就地翻转 starred 让 UI 立刻反应,backend
  返回后用最新值再次替换;失败回滚到原值 + errorMessage。
- 内部 `applyUpdated(_:)` 按 id 在 `messages` 数组里找位置整体替换 Message(struct
  全 `let`,只能整条换)。

## MessageCard 改动

引入 `@State isEditing` + `draftText` / `draftTagIds: Set<Int>` / `isSaving`。

- **只读态**:与原行为一致;右下角加一个轻量「pencil」icon,hover 才显眼(避免空闲时
  分散注意力)
- **编辑态**:
  - 正文换成 `TextEditor`(minHeight 80 / maxHeight 240,超出可滚)
  - tag chips 每个加 x 按钮,末尾加 `+` 按钮弹 popover
  - 卡片背景换淡色 + 强调边框,模式感清晰
  - 底部「取消 / 保存」按钮替换 pencil
- **星标**:两态下都是可点 Button(原来是不可点 Image)
- **未变**:actor 头像、issue、媒体网格仍只读;媒体增删 / actor 切换留给后续

## Tag popover 选择器

候选方案问用户后选了 popover 多选。`TagPickerPopover` 私有 view:
- 顶部 `TextField` 即时过滤 `availableTags`
- 列表 row 是 toggle 风格(checkmark.square),点即勾选 / 取消
- 关闭无 OK / Cancel —— 点 popover 外即消失,选择已实时同步到 `draftTagIds`
- 显示 messageCount 给用户做参考

新建 tag 走「写在正文里 #xxx 让后端自动建」—— popover 只允许选已有 tag,避免在 Mac 端
单独加 tag 创建 API。

## Save 行为细节

`saveEdit()` 比较 draft 与 message 当前值:
- 都没改 → 直接 cancelEdit,不发请求
- text 变了 / tag 没变 → 只传 text,后端会按新 text 重抽 #hashtag
- text 没变 / tag 变了 → 只传 tagIds,后端按显式 tag 替换、不解析 text
- 都变了 → 都传

最后一条注意:后端 `tag_ids != None` 时**不**再走文本 #hashtag 解析。所以同时改文字
+ tag 的语义是「我已经显式指定了 tag,以我为准」,符合直觉。

## 改动文件

- `MyNote/MyNote/APIClient.swift`:PUT → PATCH + starred 参数
- `MyNote/MyNote/MessagesViewModel.swift`:read-only 注释移除,新增 commitEdit /
  toggleStar / applyUpdated / replacing,init 加 apiClient 依赖
- `MyNote/MyNote/MessageCard.swift`:整体重写,两态 + TagPickerPopover
- `MyNote/MyNote/MessagesView.swift`:`MessageCard(message:viewModel:)` 多传 vm

## 没做的

媒体网格仍只展示 —— 增删 / 拖拽排序留给后续(已与用户对齐)。actor 切换、issue
绑定也保留只读。媒体编辑做齐之后,可能需要从 popover 升级成 sheet。
